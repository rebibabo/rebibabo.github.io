程序化广告 (4)：Ad Network 和 Ad Server——被忽略的两个重要角色

前几篇我们讲完了核心角色：

  • DSP（Demand Side Platform，需求方平台）
  • SSP（Supply Side Platform，供给方平台）
  • ADX（Ad Exchange，广告交易平台）
  • 4 个广告服务平台：CMP（程序化创意平台）、Ad Verification（广告验证）、DMP（数据管理平台）、Analytics Platform（监测分析平台）

看起来已经覆盖了整个生态。

但还有 2 个角色——它们历史悠久、至今活跃，但在新一代程序化广告语境下经常被忽略：

  • Ad Network（广告网络 / 广告联盟）：程序化广告的”前身”
  • Ad Server（广告服务器）：广告链路里的”执行引擎”

不理解它们，你对程序化广告的认识就是不完整的。

这一篇把它们讲清楚。

第一部分：Ad Network 广告网络

1.1 Ad Network 是什么

Ad Network（广告网络），中文也叫”广告联盟”。

一句话定义：

Ad Network 是一种把多个媒体的广告位”打包”卖给广告主的中介平台。

它是程序化广告的前身——
在 DSP、SSP、ADX 出现之前，Ad Network 是解决”广告主和媒体对接难题”的最早方案。

1.2 为什么会出现 Ad Network

回到 2000 年代初的互联网世界。

那时广告主和媒体的对接方式很原始：

  • 广告主直接打电话、发邮件给媒体方
  • 一份合同谈几周
  • 一次合作签几十万

问题：

  • 几千个广告主 vs 几百万个网站
  • 两两对接根本不可能
  • 中小媒体没人理（没销售团队、流量不够大）
  • 中小广告主也很苦（不知道在哪投广告）

Ad Network 出现了：

  • 把几千个中小媒体的广告位打包起来
  • 卖给广告主：”你买一次，我帮你投到 1000 个网站”
  • 解决了”多对多对接”的最早一波难题

1.3 Ad Network 怎么工作

Ad Network 的工作模式相对简单：

关键特征：

  • 通常是打包采购（不是实时竞价）
  • 价格相对固定，通常按 CPM（千次曝光成本） 或 CPC（单次点击成本） 报价
  • 广告主不知道流量具体来自哪些媒体（透明度低）
  • 媒体也不知道广告主是谁

1.4 Ad Network vs SSP / ADX：到底有什么区别

很多人会问：Ad Network 不就是 SSP/ADX 吗？

它们确实有相似之处（都是整合多媒体流量），但有本质区别：

维度 Ad Network SSP / ADX
出现时间 早（2000 年代初） 晚（2010 年后）
交易方式 打包采购 实时竞价（RTB，Real-Time Bidding）
透明度 低（看不到具体媒体） 高（每次曝光都透明）
灵活度 低（按合约执行） 高（每次出价可调）
价格机制 固定 / 协议价 动态竞价
决策频率 月级 / 周级 毫秒级
适合场景 大批量采购、品牌曝光 精准投放、效果广告

一个简单的类比：

  • Ad Network = 批发市场（一次买一大批，价格固定）
  • SSP / ADX = 股票交易所（每个商品实时拍卖，价格动态变化）

1.5 Ad Network 现在还存在吗

答案是：还存在，而且大量存在。

虽然 SSP/ADX 是新一代主流，但 Ad Network 并没有消失：

现在的 Ad Network 在做什么

  • 服务中长尾媒体：那些没有专门接 SSP 能力的小 APP、小网站
  • 提供 SDK（Software Development Kit，软件开发工具包）一键变现：媒体方接入 SDK，剩下的全交给 Ad Network
  • 打包销售：尤其是品牌广告（不需要实时竞价）

现代 Ad Network 的演化

现在的”Ad Network”已经不是 2000 年代的纯打包模式：

  • 内部融合了 SSP / ADX 的能力
  • 部分流量也支持实时竞价
  • 既能”打包卖”又能”实时竞价”

典型例子：

  • Google AdSense：表面是个 Ad Network，背后接的是 Google 的 ADX
  • 字节穿山甲：定位是”联盟”，但内部已经是个完整的 SSP/ADX 体系

1.6 真实玩家

国外：

  • Google AdSense：全球最大的广告联盟
  • Media.net：Yahoo 和 Bing 联合做的联盟
  • Taboola / Outbrain：内容推荐型联盟

国内：

  • 字节穿山甲：服务大量中小 APP
  • 腾讯优量汇：腾讯系联盟
  • 百度联盟：百度系联盟
  • 快手联盟：快手系联盟

第二部分：Ad Server 广告服务器

2.1 Ad Server 是什么

Ad Server（广告服务器） 是广告链路里的”执行引擎“。

如果说 DSP/SSP/ADX 负责”决定谁中标“，
那 Ad Server 负责”怎么展示、怎么记录“。

它是个被严重低估的角色——
几乎所有讲程序化广告的内容都会简略提一句然后跳过，
但工程实现上它是一个独立的、复杂的系统。

2.2 用一个类比理解

直接讲 Ad Server 的功能比较抽象，用一个类比先建立直觉：

Ad Server 像是广告的”私人仓库 + 调度中心”。

  • 仓库：存放所有广告创意素材（图片、视频、文案）
  • 调度中心：决定”什么时候、把哪个创意、展示给哪个用户”
  • 记录员：记录每一次展示、每一次点击、每一次转化

广告主和媒体方各自都有自己的”仓库 + 调度中心”——
这就是 Ad Server 有两种类型的原因。

2.3 Ad Server 的两种类型

Ad Server 分两类，站在谁的视角，就是谁的 Ad Server：

类型 服务对象 也叫 一句话
第三方 Ad Server 广告主 广告主侧 Ad Server 帮广告主管理跨媒体的广告投放
第一方 Ad Server 媒体方 媒体方侧 Ad Server 帮媒体方管理自己的广告位

下面分别展开。

2.4 第三方 Ad Server（广告主侧）

解决什么问题

想象一个国际大广告主——比如某汽车品牌——在全球 30 个国家、50 个媒体、3 个 DSP 上投放广告。

问题：

  • 创意素材在哪？分散在各个 DSP 后台？
  • 投放数据散落在各个媒体的报表里？
  • 同一个用户被不同媒体重复曝光怎么办？
  • 媒体方给的数据可信吗？

第三方 Ad Server 就是解决这些问题的：

  • 一个统一的中心，管理所有跨媒体的广告投放

核心功能

1. 创意素材统一管理

  • 所有创意上传到 Ad Server，统一存储
  • 投到任何媒体，都从 Ad Server 调用
  • 改一次创意，所有地方同步更新

2. 跨媒体追踪

  • 每次曝光、点击、转化都汇报到 Ad Server
  • 广告主能看到”全球 50 个媒体的统一报表”

3. 频次控制

  • 同一用户跨媒体看了几次广告？
  • 超过上限就不再展示
  • 节省广告主的钱

4. 第三方独立数据验证

  • 媒体方报的数据可能注水（”我给你曝光了 100 万次”）
  • 第三方 Ad Server 的数据是独立的
  • 作为对账依据

真实场景

宝洁、可口可乐、欧莱雅这种全球大广告主，几乎都用第三方 Ad Server——
统一管理在全球的几百个 campaign（广告投放活动）。

不用第三方 Ad Server 的话，对账、追踪、优化全部失控。

2.5 第一方 Ad Server（媒体方侧）

解决什么问题

现在换个视角——
想象你是一家大媒体（比如纽约时报）：

你的广告位收入有几个来源：

  • 直接和广告主签的合约（包年包月）
  • 通过 SSP 接入的程序化广告
  • 自己投放的内部推广
  • 公益广告（兜底）

问题：

  • 同一个广告位，有多个来源在抢，谁优先？
  • 直接合约客户优先级最高，但怎么保证执行？
  • 程序化广告什么时候介入？
  • 没人买的时候怎么兜底？

第一方 Ad Server 就是解决这些问题的：

  • 媒体方侧的”最终决策者”

核心功能

1. 广告位管理

  • 所有广告位的统一注册
  • 每个广告位的规则（屏蔽哪些行业、设置底价等）

2. 直接广告投放

  • 媒体方直接卖给广告主的合约广告
  • 不走 SSP/ADX，直接由 Ad Server 投放
  • 通常优先级最高

3. 程序化广告整合

  • 接入 SSP/ADX 的程序化广告
  • 把”合约广告 + 程序化广告”放在一起综合决策

4. 最终决策

  • 这个广告位，到底展示哪个广告？
  • 综合考虑：优先级、出价、频次、用户体验
  • Ad Server 才是”最后一步”的决策者

一个常见的误解

很多人以为”SSP 决定了展示哪个广告”——
这是错的。

正确的理解：

  • SSP / ADX 决定”程序化广告中谁中标”
  • 但媒体方还有”直接合约广告”——这些不走 SSP
  • 真正的最终决策是媒体方的 Ad Server 做的
  • 它把 SSP 返回的程序化广告 + 直接合约广告放在一起，综合决策

真实场景

像纽约时报、华尔街日报这种大媒体——
首页的广告位优先卖给奢侈品牌（直接合约），剩下的流量才走程序化。

这种”优先级管理”全靠媒体方的 Ad Server。

2.6 Ad Server 和 DSP / SSP 的关系

理清楚分工：

系统 负责什么
DSP 帮广告主在 ADX 上出价
SSP 帮媒体方把流量送到 ADX
ADX 撮合 DSP 和 SSP，决定谁中标
第三方 Ad Server（广告主侧） 跨媒体统一管理、追踪、对账
第一方 Ad Server（媒体方侧） 在媒体侧做最终展示决策

一个简化的口诀：

DSP/SSP/ADX 决定”竞价中谁赢”

Ad Server 决定”实际展示什么、记录什么”

2.7 行业演化的小知识

第三方 Ad Server 这个领域有个有趣的历史：

2000 年代初，最知名的 Ad Server 公司是 DoubleClick，它有两个产品：

  • DCM（DoubleClick Campaign Manager）：广告主侧的 Ad Server
  • DFP（DoubleClick for Publishers）：媒体方侧的 Ad Server

2008 年，Google 以 31 亿美元收购了 DoubleClick——
这是当时广告科技领域最大的收购案之一。

收购后这两个产品逐渐演化为：

  • DCM → 现在叫 Google Campaign Manager 360
  • DFP → 现在叫 Google Ad Manager（合并了 SSP/ADX 能力）

这个收购改变了整个程序化广告行业的格局——
Google 凭借 Ad Server 的统治力，逐步建立起了从 DSP（DV360）到 SSP（Google Ad Manager）的完整广告帝国。

2.8 真实玩家

第三方 Ad Server（广告主侧）：

  • Google Campaign Manager 360（原 DCM）：行业老大
  • Sizmek：曾经的第二大，2019 年被 Amazon 收购
  • Adform：欧洲市场较强
  • Flashtalking：独立第三方

第一方 Ad Server（媒体方侧）：

  • Google Ad Manager（原 DFP）：占有压倒性市场份额
  • Kevel（原 Adzerk）：API-first 的 Ad Server，被开发者喜欢
  • Equativ（原 Smart AdServer）：欧洲市场较强

国内：相对而言不像国外那么明显地区分”独立 Ad Server”。
大多数 Ad Server 能力内嵌在 DSP / SSP 系统里，
独立的 Ad Server 厂商较少。

第三部分：把这两个角色放回完整生态

现在把 Ad Network 和 Ad Server 加进我们之前画的生态全景里。

完整链路：

这张图比之前的”DSP→ADX→SSP”链路复杂多了——
但更接近真实。

关键观察：

  1. Ad Network 是一条独立的路径——和 DSP/SSP/ADX 平行存在
  2. Ad Server 在两端都存在——广告主侧和媒体方侧各一个
  3. 现代广告投放通常是多条路径并存——大广告主同时用 DSP（程序化）+ Ad Network（联盟）+ 直接合约

一些重要的认知

讲到这里，可以做一些总结性的判断：

1. Ad Network 没有死，只是定位变了

它不再是”主流交易方式”，但仍然是中长尾流量变现的最佳方案。

2. Ad Server 是广告链路里”最关键的工程系统”之一

它不出现在用户面前、不参与竞价、不抢风头，
但所有的展示决策、追踪记录、对账数据，都依赖它。

3. “竞价” 和 “执行” 是两件事

  • 竞价 = 决定谁有资格展示 → DSP/SSP/ADX
  • 执行 = 实际展示和记录 → Ad Server

理解这个分工，你就抓住了程序化广告的工程本质。

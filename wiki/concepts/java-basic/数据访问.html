数据访问

[[wiki/concepts/java-basic/Java-Basic-概念总图|返回概念总图]]

这一层回答什么问题

数据怎么存、怎么查、怎么缓存？MySQL 事务和索引怎么理解？MyBatis 和 Redis 在架构中扮演什么角色？

数据访问层是业务逻辑和数据库之间的桥梁。一条典型的查询链路是：请求 → Service → Redis 缓存 → MyBatis → MySQL。这一层关心的不是”怎么写 SQL”，而是”怎么让数据访问安全、高效、可维护”。

MySQL：持久化的基础

事务（ACID）：一组操作要么全部成功要么全部回滚。InnoDB 通过 MVCC（多版本并发控制）实现高并发事务隔离——读不阻塞写，写不阻塞读。

隔离级别：READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ（MySQL 默认）→ SERIALIZABLE。级别越高越安全，但并发性能越低。

索引：B+ 树是 InnoDB 索引的核心结构。非叶子节点只存索引键，叶子节点存完整数据（聚簇索引）或主键值（二级索引），叶子节点间用双向链表连接支持范围查询。

最左前缀原则：联合索引 (a, b, c) 中，只有查询条件从最左列开始才能利用索引。WHERE a=1 AND b=2 用索引，WHERE b=2 不用。

索引失效常见场景：对索引列用函数、LIKE 前导通配符、隐式类型转换、OR 连接非索引列。

MyBatis：SQL 与 Java 的桥梁

半自动 ORM——你写 SQL，框架帮你做结果映射。相对于 JPA/Hibernate 的全自动，MyBatis 让 DBA 和开发者对 SQL 有绝对控制权。

#{} vs ${}：#{} 使用 PreparedStatement 防 SQL 注入，${} 直接字符串拼接。能用 #{} 就不用 ${}——只有动态表名/列名这种无法预编译的场景才用 ${}。

动态 SQL：<if>、<where>、<set>、<foreach>、<choose> 等标签让你根据条件灵活拼接 SQL，而不是写死一长串。

MyBatis-Plus：MyBatis 的增强工具，BaseMapper<T> 提供通用 CRUD 无需写 XML，LambdaQueryWrapper 提供类型安全的查询条件构造。

Redis：用内存换速度

基于内存的键值存储。五种数据结构覆盖大多数场景：

数据结构 场景
String 缓存值、计数器、分布式锁
Hash 对象缓存
List 消息队列、最新列表
Set 去重、交集运算
ZSet 排行榜、延迟队列

缓存三大问题：

  • 穿透：查不存在的数据 → 布隆过滤器或缓存空值
  • 击穿：热点 key 过期 → 互斥锁或逻辑过期
  • 雪崩：大量 key 同时过期 → TTL 加随机值、多级缓存

Cache Aside 模式（最常用）：读先查缓存，未命中查 DB 后写缓存；写先更新 DB，再删除缓存（不是更新缓存——防止并发覆盖）。

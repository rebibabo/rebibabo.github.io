# 类型擦除

[[wiki/concepts/java-basic/泛型|返回泛型]]

## 这一层回答什么问题

> Java 泛型的类型参数在运行时为什么拿不到？擦除带来了什么限制？为什么 Java 选择擦除而不是真正的泛型？

类型擦除是理解 Java 泛型所有行为和限制的钥匙。它解释了为什么 `List<String>` 和 `List<Integer>` 在运行时是同一个类、为什么不能 `new T()`。

## 什么是类型擦除

编译器在编译时把泛型类型参数全部"擦除"，替换为边界类型。默认边界是 `Object`：

```java
// 你写的
public class Box<T> {
    private T content;
    public T get() { return content; }
}

// 编译后等价于
public class Box {
    private Object content;  // T 被擦除成 Object
    public Object get() { return content; }
    
    // 编译器自动插入强转代码
    // (String) box.get()
}
```

所以 `List<String>.getClass() == List<Integer>.getClass()` 返回 true——运行时它们是同一个类。

## 为什么选擦除

Java 5 引入泛型时，Java 已经有庞大的代码生态。如果做真正的泛型（像 C# 那样运行时保留类型参数），所有旧代码都得改。擦除是**向后兼容的代价**——泛型代码编译出的字节码和没泛型时完全一致。

## 擦除的 5 个后果

1. **不能 `new T()`** — 运行时不知道 T 具体是什么
2. **不能 `List<int>`** — 必须用 `List<Integer>`，基本类型不能作类型参数
3. **不能 `new T[10]`** — 泛型数组不合法
4. **不能 `instanceof List<String>`** — 运行时拿不到类型参数
5. **不能区分只差泛型参数的重载** — `void print(List<String>)` 和 `void print(List<Integer>)` 擦除后签名相同

## 桥接方法

当子类确定了泛型类型时，编译器自动生成桥接方法来保持多态：

```java
class MyList implements Comparable<MyList> {
    public int compareTo(MyList other) { ... }
}
// 编译后编译器生成桥接方法：
// public int compareTo(Object other) { return compareTo((MyList) other); }
// 没有桥接方法，多态就断了
```

## 怎么绕过擦除

当确实需要运行时获取泛型类型时（比如 Jackson 反序列化），通过子类继承的方式保留类型信息：

```java
// TypeReference 的子类在 class 文件中保留了泛型参数信息
new TypeReference<List<User>>() {};
```

这不是绕过了擦除——是绕过了"获取当前对象的泛型参数"的限制。子类在继承时确定的泛型类型会被写入 class 文件的 Signature 属性。

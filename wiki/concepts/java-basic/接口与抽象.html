接口与抽象

[[wiki/concepts/java-basic/面向对象|返回面向对象]]

这一层回答什么问题

抽象类和接口有什么区别？什么时候用哪个？接口的 default 方法解决了什么？

抽象类和接口都是定义”子类必须做什么”的契约，但适用场景不同。选错不会报错，但会让代码的扩展方向走偏。

抽象类

abstract class 是”半成品类”——不能实例化，可以包含抽象方法（没有方法体的方法）和具体方法。

为什么需要抽象类？ 当你有一组类共享同一个”是什么”的关系，且有公共状态（字段）或公共行为需要复用时。

接口

interface 是”能力契约”——定义一组行为，不关心状态。

从 Java 8 开始，接口能力大幅扩展：

default 方法的工程意义：给接口加新方法而不破坏已有实现。这是 Java 8 能在 Collection 接口里加 stream() 方法的关键——如果 stream() 是抽象方法，所有的 Collection 实现类都要改。

怎么选

抽象类 接口
关系 “is-a”（是什么） “can-do”（能做什么）
继承 只能继承一个 可以实现多个
字段 可以有实例变量 只能有常量
构造方法 可以有 没有
适用场景 共享代码和状态 定义跨层次的能力

经验法则：优先用接口。当需要共享状态或构造逻辑时，才考虑抽象类。接口定义行为边界（”你能做什么”），抽象类提供代码复用（”你已经做好了什么”）。

函数式接口

只有一个抽象方法的接口，标注 @FunctionalInterface。它可以被 Lambda 表达式替代——Runnable、Comparator、Callable 都是函数式接口。

# 类加载

[[wiki/concepts/java-basic/JVM|返回 JVM]]

## 这一层回答什么问题

> 一个类是怎么从 .class 文件变成 JVM 里的 Class 对象的？双亲委派模型是什么？为什么要打破它？

类加载是 JVM 运行的入口。不理解类加载，就不理解为什么 `ClassNotFoundException` 会在运行时出现、为什么 Spring Boot 的 Fat Jar 需要特殊处理、为什么 Tomcat 需要隔离不同 Web 应用的类。

## 类的生命周期

```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
```

**准备 vs 初始化**——这是最容易混的两个阶段：

- **准备**：为 static 变量分配内存，赋**零值**（int 是 0，boolean 是 false，引用是 null）
- **初始化**：执行 static 代码块和 static 变量的赋值语句

```java
// 准备阶段：count = 0（不是 100）
// 初始化阶段：count = 100
static int count = 100;
```

如果 `static final int` 的值是编译器常量（字面量或简单表达式），在准备阶段就确定值了——不需要等到初始化。

## 双亲委派模型

```
Bootstrap ClassLoader  ← 加载 rt.jar（java.lang.* 等核心类）
        ↑ 委派
Extension ClassLoader  ← 加载 ext/ 目录
        ↑ 委派
Application ClassLoader ← 加载 classpath
```

**工作流程**：一个类加载器收到加载请求 → 先问父加载器能不能加载 → 父能加载就不自己加载 → 父不能加载才自己加载。

**好处**：`java.lang.String` 永远由 Bootstrap 加载器加载，你写一个自定义的 `String` 类不会被加载——防止核心类被篡改。

## 什么时候打破双亲委派

JDBC 是一个经典例子。`DriverManager` 是 Bootstrap 加载的，但它需要加载 classpath 上的 `com.mysql.cj.jdbc.Driver`——而 Bootstrap 加载器看不到 classpath。所以引入**线程上下文类加载器**（Thread Context ClassLoader）来打破委派。

Tomcat 也需要打破——不同 Web 应用可能依赖不同版本的同一个 jar，必须隔离各自的类加载器。

## 触发初始化的条件

不是所有类都在 JVM 启动时加载。只有以下情况才触发初始化：
- `new`、读写 static 字段、调用 static 方法
- 反射调用
- 初始化子类时父类先初始化
- 启动时包含 main 方法的类

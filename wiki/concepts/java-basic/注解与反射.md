## 注解：代码的元数据

注解本质是一个特殊的接口（`@interface`），本身不做事——需要有人（反射、APT、编译器）去读它。

**元注解控制注解的行为范围：**

| 元注解 | 含义 |
|--------|------|
| `@Target` | 注解能加在哪（类/方法/字段/参数...） |
| `@Retention` | 注解保留到什么时候（SOURCE/CLASS/RUNTIME） |
| `@Documented` | 是否出现在 Javadoc 中 |
| `@Inherited` | 子类是否继承父类的注解 |

`@Retention(RUNTIME)` 是 Spring 能用反射读取注解的前提——编译期和 CLASS 级别的注解运行时不可见。

## 反射：运行时操作类和对象

正常写代码是编译期确定一切。反射反过来——运行时拿到一个类名，就能创建对象、调用方法、读写字段。

核心 API：

```java
Class<?> clazz = Class.forName("com.example.User");

// 创建对象
Object obj = clazz.getDeclaredConstructor().newInstance();

// 读写字段（包括 private）
Field field = clazz.getDeclaredField("name");
field.setAccessible(true);
field.set(obj, "张三");

// 调用方法
Method method = clazz.getDeclaredMethod("sayHello");
method.invoke(obj);
```

`getDeclaredXxx` 拿本类声明的（含 private），`getXxx` 拿 public 的（含继承的）——这个规律贯穿 Constructor、Field、Method 三类 API。

## 动态代理：AOP 的底层

运行时动态生成一个实现指定接口的代理类。所有方法调用首先经过 `InvocationHandler`：

```java
InvocationHandler handler = (proxy, method, args) -> {
    System.out.println("before");       // 前置增强
    Object result = method.invoke(target, args);
    System.out.println("after");        // 后置增强
    return result;
};
MyService proxy = (MyService) Proxy.newProxyInstance(
    loader, new Class[]{MyService.class}, handler);
```

这就是 `@Transactional` 和 `@Cacheable` 的底层——Spring 给你的是代理对象，你在代理上调方法，代理在前后加事务/缓存逻辑。

**关键坑：内部调用 AOP 失效**

```java
public void methodA() {
    this.methodB();  // ❌ this 是原始对象，不是代理
}
```

`this` 调用绕过了代理，所以 `@Transactional` 注解在 `methodB` 上不生效。解决方案：注入自身，或拆到不同类。

## 它们和 Spring 的关系

Spring 的工作流就是注解 + 反射 + 动态代理的组合：

```
@Component                    → 反射扫描，创建 Bean 放入容器
@Autowired                    → 反射读写字段，从容器找依赖注入
@Transactional                → 动态代理包裹方法，前后加事务逻辑
@Value("${app.name}")         → 反射读取注解属性值，从配置文件取值注入
```

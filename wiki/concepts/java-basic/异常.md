# 异常

[[wiki/concepts/java-basic/Java-Basic-概念总图|返回概念总图]]

## 这一层回答什么问题

> 程序出错了怎么办？Checked 和 Unchecked 异常有什么区别？try-with-resources 为什么能自动关资源？

异常不只是"错误处理"——它是一种控制流机制。用好了，代码干净、问题可追溯；用不好，吞异常、丢堆栈、生产环境排错全靠猜。

## 异常的类层次

```
Throwable
├── Error（不应捕获：OOM、StackOverflow）
└── Exception
    ├── RuntimeException（Unchecked，不强制处理）
    │   ├── NullPointerException
    │   ├── IllegalArgumentException
    │   └── IndexOutOfBoundsException
    └── 其他 Exception（Checked，编译器强制处理）
        ├── IOException
        └── SQLException
```

## Checked vs Unchecked

| 维度 | Checked | Unchecked |
|------|---------|-----------|
| 父类 | Exception（非 RuntimeException） | RuntimeException |
| 编译检查 | 必须 try-catch 或 throws | 不强制 |
| 典型场景 | 外部条件导致的异常（IO、SQL） | 程序 bug（NPE、数组越界） |
| 设计意图 | 调用者有义务处理 | 应该通过修代码避免 |

经验法则：能用代码逻辑避免的（判 null、判边界），不要用异常；外部条件不可控的（网络、文件），用 Checked Exception 让调用者知道。

## try-with-resources

Java 7 开始，实现了 `AutoCloseable` 的资源可以自动关闭：

```java
// 传统写法：finally 关资源，嵌套 ugly
// Java 7+：自动 close，即使抛异常也能正确关闭
try (BufferedReader br = new BufferedReader(new FileReader(path))) {
    return br.readLine();
}
// br.close() 自动调用，不需要 finally
```

## 异常链

抛出新异常时把原始异常作为 cause 传进去：

```java
try {
    // ...
} catch (SQLException e) {
    throw new ServiceException("查询用户失败", e);  // e 作为 cause
}
```

关键意义：上层看到 `ServiceException`，但排查时能一路追到 `SQLException` 的根因。

## 常见反模式

- **吞异常**：`catch (Exception e) {}` — 什么都没做，出问题无从排查
- **用异常控制流程**：正常业务流程应该用 if-else，不要 throw-catch
- **finally 里 return**：会吞掉 try 块抛出的异常

构建工具

[[wiki/concepts/java-basic/Java-Basic-概念总图|返回概念总图]]

这一层回答什么问题

你的项目是怎么从源码变成可运行的 jar 的？依赖怎么管理、冲突怎么解决？Maven 和 Gradle 怎么选？

构建工具是把源码变成可运行产物的桥梁。它不只是”点一下编译”——它管理依赖、控制打包、运行测试、处理多模块项目。不理解构建工具，就理解不了一个 Java 项目的完整生命周期。

这层的主要分支

  • [[wiki/concepts/java-basic/Maven|Maven]] — pom.xml、GAV 坐标、依赖 scope、生命周期
  • [[wiki/concepts/java-basic/Gradle|Gradle]] — 增量编译、implementation vs api、Version Catalog

Maven

基于 pom.xml 的声明式构建工具。核心概念：

GAV 坐标：groupId:artifactId:version 三者唯一确定一个依赖。

依赖范围（Scope）：

scope 编译 测试 运行时 例子
compile（默认） ✓ ✓ ✓ spring-core
provided ✓ ✓ ✗ servlet-api（容器提供）
runtime ✗ ✓ ✓ mysql-connector
test ✗ ✓ ✗ junit

依赖传递与冲突：A 依赖 B，B 依赖 C → A 自动依赖 C。当 A→C(v1) 和 B→C(v2) 同时存在时，最短路径优先——离根近的版本胜出。显式 <exclusion> 可以排除不需要的传递依赖。

生命周期：validate → compile → test → package → verify → install → deploy。后一个阶段会自动执行前面所有的阶段。

Gradle

基于 Groovy/Kotlin DSL 的构建工具，相对 Maven 的优势：

  • 增量构建：只编译改过的部分
  • 构建缓存：跨项目复用编译结果
  • 灵活性：DSL 比 XML 表达能力更强

Gradle Wrapper (gradlew) 保证团队所有人用同一版本 Gradle，不需要手动安装。

版本管理进阶

BOM（Bill of Materials）：统一管理一组依赖的版本，子模块只需引入依赖不需要写版本号。Spring Boot 的 spring-boot-dependencies 就是一个 BOM。

Version Catalog（libs.versions.toml）：Gradle 7+ 的集中版本管理方式，一个文件管理所有依赖版本，多模块共享。

Maven vs Gradle 选型

Maven Gradle
配置 XML（啰嗦但明确） DSL（灵活但学习曲线）
速度 每次全量 增量+缓存
生态 极其成熟 Spring 官方已迁移
推荐 遗留项目、简单项目 新项目、多模块项目

注解和反射是几乎所有 Java 框架的底层积木。理解它们不是为了手写框架，而是为了让框架在你脑子里从”魔法”变成”可以跑通的流程”。

注解：代码的元数据

注解本质是一个特殊的接口（@interface），本身不做事——需要有人（反射、APT、编译器）去读它。

元注解控制注解的行为范围：

元注解 含义
@Target 注解能加在哪（类/方法/字段/参数…）
@Retention 注解保留到什么时候（SOURCE/CLASS/RUNTIME）
@Documented 是否出现在 Javadoc 中
@Inherited 子类是否继承父类的注解

@Retention(RUNTIME) 是 Spring 能用反射读取注解的前提——编译期和 CLASS 级别的注解运行时不可见。

反射：运行时操作类和对象

正常写代码是编译期确定一切。反射反过来——运行时拿到一个类名，就能创建对象、调用方法、读写字段。

核心 API：

getDeclaredXxx 拿本类声明的（含 private），getXxx 拿 public 的（含继承的）——这个规律贯穿 Constructor、Field、Method 三类 API。

关键坑：内部调用 AOP 失效

this 调用绕过了代理，所以 @Transactional 注解在 methodB 上不生效。解决方案：注入自身，或拆到不同类。

执行模型

[[wiki/concepts/concurrency/并发总图|返回并发总图]]

定义

这一页关注的是：

Java 程序从源码到线程真正跑在 CPU 上，中间发生了什么？

它是并发系列最底的一层地基。

这一层包含什么

  • 程序如何从源码一路变成最终在 CPU 上执行的机器指令
  • JVM、操作系统、CPU 在执行链路里分别负责什么
  • CPU 核心、寄存器、缓存怎样影响执行速度和并发行为
  • 线程如何被操作系统调度到不同 Core 上运行
  • Heap、Stack、对象、引用之间是什么关系
  • 为什么“共享的、会变的对象”会天然引出线程安全问题

一条最重要的执行链

并发问题不能只从 Java 语法表面看，它实际发生在下面这条链路里：

可以把它先记成 5 步：

  1. 写出 Java 源码。
  2. javac 把源码编译成字节码。
  3. JVM 加载字节码并执行，必要时把热点路径编译成更贴近硬件的机器指令。
  4. 操作系统负责创建和调度线程。
  5. CPU 最终在某个 Core 上执行这些指令。

后面的 count++、synchronized、CAS、volatile 都只是这条链路上的不同约束方式。

JVM、操作系统、CPU 分别做什么

  • JVM 负责加载类、执行字节码、管理 Heap / Stack，并把 Java 线程映射到系统线程。
  • 操作系统负责线程调度，决定哪个线程此刻能拿到 CPU 时间片。
  • CPU 只执行最终机器指令，它不理解“Java 语义”，只理解寄存器、缓存、读写和跳转。

这也是为什么很多并发问题不能只问“Java 代码看起来是不是顺序的”，而要继续问：

  • 线程什么时候会被切走
  • 数据究竟在寄存器、缓存还是主内存里
  • 两个线程是否真的在同时读写同一份对象状态

CPU 这一层真正发生了什么

一个 CPU 不是一个单独的执行点，而是由多个 Core 组成。单个 Core 在同一时刻通常只能执行一个线程，其余线程只能等待调度。

Core 为了加快执行，会优先使用：

  • Register：保存当前参与计算的少量数据，速度最快
  • Cache：缓冲最近会访问的数据，减少每次都去主内存取数的开销
  • Main Memory：共享主内存，容量更大但更慢

所以一条看似简单的语句，例如 int c = a + b，并不是“源码写完就结束了”，而是会变成读取、装载、计算、写回这一串具体动作。

为什么多核不会自动让程序变快

现代 CPU 主要靠增加 Core 数量来提升整体吞吐，但这不代表单线程代码会自动并行。

如果程序本身只有一条顺序执行路径，那么再多 Core 也只是让其中一个 Core 在工作。想真正利用多核，必须把任务拆成多条线程，由操作系统调度到不同 Core 上执行。

这一步一旦发生，就会立刻引出两个并发基础事实：

  • 同一时刻可能有多个线程同时访问同一份共享对象
  • 线程执行先后不再由源码表面顺序唯一决定

Heap、Stack、对象、引用

这一层如果不先看清，后面的线程安全问题会显得很抽象。

  • Stack 通常是线程私有的，保存局部变量、方法栈帧等
  • Heap 通常是线程共享的，对象实例大多放在这里
  • 局部变量本身也许在线程自己的栈里，但它保存的引用可能指向 Heap 里的共享对象

所以“变量写在方法里”不等于“它一定线程安全”。真正要看的是：

  • 这个变量是不是只在当前线程内使用
  • 它引用的对象是不是被多个线程共同访问
  • 这个对象的状态会不会被修改

为什么会自然走向并发问题

并发 bug 的根源不是“用了线程”本身，而是：

多个线程同时访问同一份共享且可变的状态。

一旦多个线程都能读写同一个 Heap 对象，它们的执行就会受到调度时序、寄存器暂存、缓存副本和写回顺序的共同影响。

count++ 会出错，只是这个底层事实第一次暴露出来的地方。

它为什么重要

后面的 count++、volatile、JMM、synchronized 这些问题，本质上都不是孤立 API 问题，而是执行模型的直接后果。

如果不先理解这一层，并发知识很容易变成背结论。

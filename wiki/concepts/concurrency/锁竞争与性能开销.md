# 锁竞争与性能开销

[[wiki/concepts/concurrency/同步机制|返回同步机制]]

## 定义

这一页关注的是：

> `synchronized` 为什么能保证安全，但又为什么可能拖慢程序？

## 它解决什么问题

它把“锁带来的代价”拆成更清楚的几层：

- 竞争失败要等待
- 可能发生阻塞与唤醒
- 线程切换会带来额外成本
- 锁粒度过大时，会把本来能并行的工作也串行化

## 先区分两种情况

### 无竞争

如果线程进入同步区域时，锁刚好是空闲的，它可以直接拿到锁并执行临界区。

这时虽然仍然有加锁和解锁动作，但没有其他线程排队等待，成本通常比较可控。

### 有竞争

如果线程尝试进入同步区域时，锁已经被其他线程持有，它就只能等待。

真正拖慢程序的，往往不是 `synchronized` 这几个字符，而是竞争之后带出来的一整串连锁反应。

## 竞争时线程会发生什么

可以先用一个简化模型理解：

1. 线程尝试获取锁
2. 锁空闲，进入临界区
3. 或者锁被占用，当前线程无法继续执行受保护代码
4. 线程进入等待竞争、阻塞或再次调度的流程

在 Java 线程状态里，等待进入 `synchronized` 区域的线程会出现 `BLOCKED` 这一类状态。

这说明一件很关键的事：

> 竞争失败的线程不是“慢一点继续干”，而是根本进不了临界区。

## 为什么它重要

并发优化常见误区不是“不知道怎么加锁”，而是：

- 锁加对了，但加太大了
- 锁能保证正确性，但吞吐量下降很明显

所以这一页是从“正确性”走向“性能权衡”的入口。

## 成本主要从哪里来

### 阻塞和唤醒

如果线程长时间拿不到锁，让它一直占着 CPU 空转并没有意义。系统通常会让它阻塞，等锁释放后再唤醒。

这个过程中会产生额外动作：

- 线程暂停执行
- 锁释放后等待线程被唤醒
- 被唤醒线程重新进入可运行队列
- 再次参与锁竞争

这些步骤都不是业务逻辑本身，却会消耗时间。

### 上下文切换

CPU Core 同一时刻只能执行一个线程。线程从 A 切到 B 时，系统需要保存和恢复执行现场，这就是上下文切换。

锁竞争会增加这类切换的概率，因为：

- 一个线程持锁执行
- 其他线程等待
- 锁释放后等待线程重新被调度

线程越多、竞争越激烈，这部分调度成本越明显。

### 临界区过大

如果把网络 IO、磁盘 IO、复杂计算都包进同一把锁里，线程持锁时间就会被拉长。

这样一来，其他线程等待的不是“那几行真正需要互斥的状态更新”，而是整个大方法。

所以优化锁，不是简单地“少用锁”，而是：

- 保证一次完整业务操作仍然是原子的
- 同时尽量缩小真正需要互斥的那部分临界区

## 什么时候该想到它

如果代码已经正确，但出现下面这些现象，就应该回到这一页看：

- 线程数增加了，吞吐量却没有上去
- CPU 忙，但很多线程在等待
- 某个同步方法很短看起来没问题，但高并发下明显变慢
- 多个不相关操作被同一把锁串行化了

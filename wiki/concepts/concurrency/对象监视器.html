Object Monitor

[[wiki/concepts/concurrency/同步机制|返回同步机制]]

定义

Object Monitor 是 Java 对象锁和 wait / notify 语义背后的监视器机制。

它解决什么问题

它回答的是：

为什么 synchronized 不只可以做互斥，还能配合 wait() / notify() 完成线程等待与通知？

Monitor 不是普通对象字段

写 synchronized (lock) 时，线程竞争的不是 lock 这个对象里某个业务字段，而是它关联的监视器语义。

可以先粗略把 Monitor 理解成 JVM 为对象维护的一组状态：

  • 当前锁是否空闲
  • 当前由哪个线程持有
  • 重入了多少次
  • 哪些线程在等这把锁
  • 哪些线程已经 wait() 在这个对象上

这也是为什么 Object Monitor 本质上是在解释：

一个对象为什么既能当互斥锁，又能当等待通知的挂点。

为什么必须用同一个锁对象

只有多个线程竞争的是同一个对象关联的 Monitor，互斥和等待通知才会真的发生在同一套机制上。

如果：

  • 线程 A 在 lock1 上等待
  • 线程 B 却在 lock2 上通知

那么它们根本不在同一个 Monitor 语义里，通知自然不会生效。

所以这里比“都是 synchronized 就行”更重要的是：

必须是同一个对象。

等待拿锁，和 wait() 不是一回事

这两个状态很容易混淆。

第一类线程：

  • 还没进入 synchronized
  • 只是因为对象锁被别人占着，所以进不去

第二类线程：

  • 已经成功进入了 synchronized
  • 但因为业务条件不满足，主动调用 wait()

这两者等待的完全不是同一件事。

原文里很重要的一点是，可以把 Monitor 大致想成有不同等待区域：

  • 等对象锁的线程
  • 已经 wait() 等通知的线程

wait() 的核心语义是什么

wait() 最关键的不是“暂停线程”，而是：

在持有对象锁的前提下，释放这把锁，并进入等待通知的集合。

这一步非常关键，因为只有等待线程把锁让出来，其他线程才有机会进入同步块，去修改它所等待的条件。

所以 wait() 和普通 sleep() 的根本区别之一就在这里：

  • sleep() 不释放当前持有的对象锁
  • wait() 会释放当前对象锁，并等待后续通知

为什么 wait/notify 必须写在 synchronized 里

如果不在持有这把对象锁的情况下调用 wait() / notify() / notifyAll()，Java 会直接抛 IllegalMonitorStateException。

背后的原因不是语法约束，而是为了保证下面几步必须围绕同一个 Monitor 完成：

  • 检查条件
  • 进入等待
  • 修改条件
  • 发出通知

否则就会出现经典的丢失通知问题：线程还没真正进入等待，通知已经先发出去了。

notify() 为什么不等于“立刻继续运行”

notify() 做的事只是：

从等待通知的线程里挑一个，让它有资格回来重新竞争对象锁。

它不表示：

  • 锁已经交过去
  • wait() 立刻返回
  • 条件已经百分之百满足

因为通知线程调用 notify() 时，自己通常还在同步块里，锁还没有释放。等待线程得等它退出同步块后，重新竞争成功，wait() 才会真正返回。

为什么也必须用 while 检查条件

和 Condition 一样，Object Monitor 这条线也应该用：

而不是 if。

原因同样是：

  • 收到通知不等于条件现在一定满足
  • 被通知线程要重新竞争锁
  • 它重新拿到锁时，条件可能又变了

所以正确语义始终是“被通知后回来再检查一次”。

它和锁对象的关系

  • 锁对象是 Java 代码里写出来的对象
  • Monitor 是 JVM 用来管理这把对象锁和等待线程的机制

所以线程竞争的不是“对象字段本身”，而是这个对象关联的 Monitor。

为什么它重要

理解 Monitor 之后，很多语言级并发语义会更清楚：

  • synchronized 锁的到底是什么
  • 为什么 wait() 必须在持锁状态下调用
  • 为什么 notify() 之后线程还要重新竞争锁
  • 为什么 wait() 和“等锁”不是同一件事

在系列里的位置

它是内置锁模型的重要总入口，和 [[wiki/concepts/concurrency/ReentrantLock|ReentrantLock]] + Condition 构成两条平行但相似的等待通知体系。

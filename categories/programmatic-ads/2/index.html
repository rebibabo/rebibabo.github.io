程序化广告 (1)：DSP——广告主的"代理人"

上一篇讲了程序化广告的全景，这一篇深入第一个核心角色：DSP（Demand Side Platform，需求方平台）。

DSP 是整个程序化广告生态里最早成熟、商业模式最清晰的角色。理解了 DSP，就理解了”广告主怎么用算法和数据把钱花得更准”。

DSP 的目的是什么

一句话：

DSP 的目的是帮广告主用最少的钱，把广告投给最对的人，获得最好的效果。

它服务的对象是需求方——也就是广告主（出钱投广告的人）。

具体来说，DSP 要替广告主做 4 件事：

  1. 触达对的人：从海量流量里找出”目标用户”
  2. 在对的时机出价：判断”这次曝光值不值得我出高价”
  3. 优化 ROI：广告主投 1 元，希望收回 3 元——DSP 的算法持续优化”每一分钱花在哪最划算”
  4. 跨平台覆盖：广告主不想同时管 100 个媒体，DSP 一次投放自动分发

类比：DSP 就像广告主的”AI 投资顾问”——你给我钱，我帮你判断”什么时候投、投多少、投给谁”。

一个具体场景

为了让 DSP 的工作方式更直观，看一个真实场景：

广告方：一家电商公司（假设叫”星辰电商”），想在双 11 推广新款蓝牙耳机。

  • 目标用户：25-35 岁、对数码产品感兴趣的男性
  • 预算：100 万人民币
  • 目标：每个有效点击成本不超过 3 元（CPC ≤ 3 元）

星辰电商的选择：

  • 方案 A：自己组建 50 人技术团队，对接 100 个媒体——成本太高、周期太长
  • 方案 B：把广告投放交给一个 DSP——立刻可用、按效果付费

显然选方案 B。

DSP 接到任务后做什么：

  1. 把”目标用户特征”录入系统
  2. 在用户访问任何接入的媒体时，实时判断这个用户符不符合特征
  3. 符合 → 出价竞争这次曝光机会
  4. 不符合 → 跳过，钱留到下一个对的人

整个过程 每秒发生几十万次——这就是程序化广告”实时竞价”的本质。

QPS 是什么

刚才提到”每秒几十万次”，这里要先解释一个概念：

QPS（Queries Per Second，每秒查询数）

它衡量一个系统”每秒能处理多少次请求”。

举几个直观的对比：

系统类型 典型 QPS
个人博客 10-100
中小企业网站 1,000-10,000
普通电商网站 10,000-100,000
大型 DSP 系统 100,000-1,000,000+
微信 / 抖音核心服务 1,000,000+

为什么 DSP 需要这么高的 QPS？

因为每次用户打开一个网页或 APP，所有接入的 DSP 都会被询问”你出多少价”，而且必须在 100 毫秒内回复。

一个大型 DSP 可能接入几千个媒体，每秒被询问几十万次——
这就是 DSP 系统的性能挑战。

DSP 由什么构成

DSP 内部是一个相当复杂的系统。核心模块有：

1. 竞价服务器（Bid Server）

DSP 的”心脏”。

  • 接收 AdX/SSP 发来的竞价请求
  • 100 毫秒内做出”是否出价、出多少价”的决策
  • 处理几十万 QPS 的高并发

2. 用户画像系统

DSP 自己积累的用户标签库。

  • 记录用户在各个媒体的行为
  • 给每个用户打标签：兴趣、年龄、性别、消费能力
  • 数据规模通常是几亿用户、几百亿条行为

3. 广告主管理后台

广告主用来”操作 DSP”的界面。

  • 上传广告创意（图片、视频、文案）
  • 设置定向条件（投给谁）
  • 设置预算和出价策略
  • 查看效果报表

4. 召回与排序引擎

决定”这次曝光给广告主 A 还是广告主 B 的广告”。

  • 召回：从几万条广告中找出 1000 条相关的
  • 排序：用 CTR（Click-Through Rate，点击率） 预估模型选出最值得展示的 1 条

5. 出价策略

决定”出多少钱合适”。

  • 太低 → 抢不到曝光
  • 太高 → 浪费预算
  • 用算法找到”最佳出价”

6. 数据回流系统

收集”投放后发生了什么”。

  • 用户看了广告吗？（曝光）
  • 点击了吗？（点击）
  • 买东西了吗？（转化）
  • 这些数据反过来优化下次出价

7. 报表系统

给广告主看”钱花得怎么样”。

  • 实时数据看板
  • 多维度分析（按时间、地域、人群）
  • 效果报告

DSP 给广告主提供什么

广告主选择 DSP 时，主要关心 6 个维度：

1. 流量（Inventory）

DSP 能触达多少媒体的流量？

  • 接入的 AdX/SSP 数量
  • 覆盖的用户规模（DAU 日活、MAU 月活）

DAU = Daily Active Users，日活跃用户
MAU = Monthly Active Users，月活跃用户

  • 是否覆盖目标人群所在的地域

广告主的核心诉求：流量越大、越精准，越好。

2. 产品（Product）

DSP 提供哪些产品形态？

  • 展示广告（Banner）
  • 信息流广告（Native）
  • 视频广告（Video）
  • 激励视频广告（Rewarded Video）
  • 开屏广告（Splash）

广告主的核心诉求：根据投放目标选合适的形态。

3. 收费方式（Pricing Model）

DSP 怎么向广告主收钱？

模式 全称 中文 说明
CPM Cost Per Mille 千次曝光成本 按”广告展示了 1000 次”收费
CPC Cost Per Click 单次点击成本 按”用户点击了广告”收费
CPA Cost Per Action 单次行动成本 按”用户完成了下载/购买”收费
oCPM Optimized CPM 优化千次曝光成本 按 CPM 收费，但用算法优化转化
oCPC Optimized CPC 优化单次点击成本 按 CPC 收费，但用算法优化转化

广告主的核心诉求：付费方式和投放目标对齐——
拉新看 CPA，曝光看 CPM。

4. 效果（Performance）

DSP 实际带来的投放效果如何？

  • CTR（Click-Through Rate，点击率）
  • CVR（Conversion Rate，转化率）
  • ROI（Return on Investment，投资回报率）
  • ROAS（Return on Ad Spend，广告支出回报率）

广告主的核心诉求：效果是检验 DSP 的唯一标准。

5. 服务类型（Service Type）

这是最关键的一个维度——决定了广告主”用哪种方式使用 DSP”。

主要有三种模式：

图片来源：《程序化广告：个性化精准投放实用手册》梁丽丽 著

模式 全称 中文 使用方式 附加服务
SaaS Software-as-a-Service 软件即服务 产品使用 贴牌
PaaS Platform-as-a-Service 平台即服务 API 接口 个性化定制
IaaS Infrastructure-as-a-Service 基础设施即服务 本地部署 提供源码

三者的差异：

SaaS：开箱即用

  • 广告主直接用 DSP 提供的产品界面
  • 只能用”投放”功能，看不到数据和技术细节
  • 可以贴牌（白标）：广告主可以把 DSP 的产品改成”自家品牌”对外用

适合：中小广告主、不需要深度定制的客户。

PaaS：API 接入

  • 通过 API 把 DSP 集成到自己的系统里
  • 可以拿到投放和数据权限
  • 支持个性化定制

适合：大广告主、有自己技术团队、需要把 DSP 接入内部系统的客户。

IaaS：本地部署

  • DSP 提供源码，广告主自己部署一套
  • 完全控制投放、数据、技术
  • 数据不出广告主的服务器

适合：大型广告主、对数据安全极度敏感的客户（如金融、政府）。

记忆方法：

SaaS = 我直接用你的产品

PaaS = 你给我 API，我集成

IaaS = 你给我源码，我自己建

Trading Desk 是什么，和 DSP 什么关系

讲完 DMP，再讲另一个容易混淆的角色：Trading Desk（交易桌）。

Trading Desk 解决什么问题

大广告主（比如可口可乐、宝洁）通常会同时使用 多个 DSP：

  • DSP A 在腾讯系流量上效果好
  • DSP B 在字节系流量上效果好
  • DSP C 在海外流量上效果好

如果分别管理 3 个 DSP——
要登录 3 个后台、看 3 套报表、配 3 套预算——效率极低。

Trading Desk 就是解决这个问题的：

Trading Desk 是一个整合多个 DSP 的统一工作台。

广告主在 Trading Desk 上：

  • 统一管理预算
  • 统一看报表
  • 统一调整策略
  • Trading Desk 自动分发到各个 DSP 去投放

DSP 提供哪些 API 给 Trading Desk

为了让 Trading Desk 能”统一管理”，DSP 通常对外提供两类 API：

1. Report API（报表 API）

让 Trading Desk 能拉取 DSP 的投放数据。

  • 曝光数、点击数、转化数
  • 花费、CTR、CVR、ROI
  • 按时间、人群、地域的多维度数据

2. Operation API（操作 API）

让 Trading Desk 能远程操控 DSP。

  • 创建/修改/暂停广告计划
  • 调整出价、调整预算
  • 上传/管理创意素材
  • 设置/修改定向条件

Trading Desk 的面板信息

一个典型的 Trading Desk 面板会展示：

整体维度：

  • 总预算 / 已花费 / 剩余预算
  • 总曝光 / 总点击 / 总转化
  • 平均 CTR / CVR / ROI

分 DSP 维度：

  • 每个 DSP 的预算分配
  • 每个 DSP 的效果对比
  • 哪个 DSP 在哪种场景下效果最好

分人群维度：

  • 不同人群的效果
  • 不同地域的效果
  • 不同时段的效果

分创意维度：

  • 不同创意的 CTR 对比
  • 哪个创意最值得加大投放

一句话总结 Trading Desk

Trading Desk = 多个 DSP 的”中控台”

它本身不投广告，而是整合 + 协调 + 优化多个 DSP 的投放

真实世界的 DSP 例子

最后给几个真实的 DSP，让你感受这个行业：

国外

  • The Trade Desk：全球最大的独立 DSP，市值数百亿美元
  • Google DV360：Google 旗下，覆盖 Google 全系流量
  • Amazon DSP：亚马逊旗下，电商场景最强

国内

  • 巨量引擎：字节旗下，覆盖抖音、今日头条等
  • 腾讯广告：腾讯旗下，覆盖微信、QQ、腾讯视频等
  • 阿里妈妈：阿里旗下，覆盖淘宝、天猫等

有趣的事实：

  • 国外 DSP 大多是”独立第三方”（如 The Trade Desk）
  • 国内 DSP 大多是”巨头自营”（依附于自家流量）
  • 这个差异背后是中美互联网生态的不同

下一篇预告

讲完了 DSP 本身，下一篇我们看 DSP 身边的”4 个帮手”——

  • CMP（Creative Management Platform，程序化创意平台）：管理和优化创意
  • Ad Verification（广告验证平台）：验证广告真实展示
  • DMP（Data Management Platform，数据管理平台）：管理用户画像数据
  • Analytics Platform（监测分析平台）：分析投放效果

这 4 个角色合起来，让广告投放从”能用”变成”好用”。

Java高并发底层原理（二十九）—— 从内存队列到可靠任务系统：数据库任务表与 MQ 如何选择

上一章讨论的是单机内存版任务处理系统：请求线程把 Task 放入有界队列，Worker 从队列中取出任务并执行。这个模型可以解决任务异步化、短暂削峰、有限并发执行的问题，但它还有一个明显边界：任务只存在于 JVM 内存里。

一旦服务重启、机器宕机，或者系统部署成多个实例，内存队列就很难继续保证任务可靠。文件解析任务如果只放在本地队列中，进程一退出，队列里还没执行的任务就会消失；Worker 正在执行到一半的任务，也没有稳定位置记录它到底完成了没有。

本章继续沿用文件解析任务这个例子，讨论任务不能丢时，系统为什么要从内存队列升级到数据库任务表或 MQ。重点不是直接比较两个工具，而是先看内存队列缺了什么，再看数据库任务表和 MQ 分别补上了哪一部分能力，以及它们各自的边界。

1、为什么内存队列无法保证任务可靠

上一章的任务提交流程是这样的：

所以内存队列的边界可以从三个问题理解：

问题 内存队列的表现
服务重启后任务还在吗 不在，队列随进程消失
任务执行到哪一步了 不知道，状态没有稳定记录
多台机器能共享同一个队列吗 不能，每个 JVM 只有自己的本地队列

因此，只要文件解析任务具备“必须最终处理”的要求，就不能只依赖内存队列。此时需要引入一个更稳定的位置保存任务，让任务即使离开当前 JVM 也能被找回来。

2、数据库任务表如何让任务可恢复

数据库任务表要解决的第一个问题是：任务不能只存在于内存里，必须有一个可恢复的状态记录。

它不是简单地把 BlockingQueue 换成数据库，而是让数据库成为任务的事实来源。内存队列仍然可以存在，但它只负责加速本机 Worker 取任务；真正判断任务有没有创建、有没有成功、是否需要重试，要看数据库任务表。

DEAD 留给人工排查或补偿

数据库任务表解决的不是高吞吐，而是可恢复。它让系统从“任务在内存里排队”升级成“任务有稳定状态，服务重启后还能继续处理”。

所以数据库任务表不是只建一张表就结束了。只要进入多实例场景，就必须同时考虑状态抢占、超时恢复和业务幂等。

4、什么时候需要 MQ：从任务状态管理到任务投递通道

数据库任务表能让任务可恢复，也能通过状态抢占支持多实例处理。但它并不是专门为高吞吐任务投递设计的。

如果文件解析任务量不大，数据库任务表很合适。因为它查询方便，状态清晰，失败任务也容易排查。但如果任务量很大，比如每秒都有大量文件处理、通知、同步任务产生，数据库就会承担频繁插入、扫描、状态更新的压力。

这时需要思考另一个问题：系统真正需要的，可能不只是“保存任务状态”，而是一个能承接大量任务投递、支持多消费者消费、失败后可以重投的通道。

这个通道就是 MQ。

MQ 的角色关系是：

在文件解析任务里，上传服务作为 Producer，把文件解析消息发送到 MQ；MQ 的 Broker 负责保存和投递消息；解析服务作为 Consumer 消费消息并处理文件。

和数据库任务表相比，MQ 更强调“投递与消费”：

关注点 数据库任务表 MQ
核心能力 保存任务状态 分发任务消息
消费方式 定时扫描或抢占 Broker 主动投递或消费者拉取
扩展方式 多实例抢任务 增加消费者实例
失败处理 自己实现重试和死信 通常内置重投和死信能力
查询排查 直接查表方便 通常要配合日志和业务表

因此，MQ 不是“更高级的数据库任务表”。它们解决的问题侧重点不同：数据库任务表更像任务状态中心，MQ 更像分布式任务投递通道。

当系统主要矛盾是“任务必须可查、可补偿、状态要清晰”时，数据库任务表更自然；当主要矛盾是“任务量大、需要削峰、多消费者扩展”时，MQ 更合适。

但“处理成功后再 ack”也带来另一个边界：如果业务处理成功了，但 ack 之前 Consumer 宕机，MQ 可能会再次投递同一条消息。因此 MQ 消费通常要默认面对“至少一次投递”，也就是消息可能重复到达。

死信队列不是垃圾桶。它的作用是把长期失败的消息从正常消费链路中隔离出来，避免它一直阻塞正常消息，同时保留排查和人工补偿入口。

MQ 场景下，几个概念要放在同一条链路里理解：

概念 在流程中的作用
Broker 保存并投递消息
Consumer 接收消息并执行业务
Ack 告诉 Broker 消息已完成
Retry 临时失败后重新投递
Dead Letter 多次失败后隔离
Idempotency 兜底重复投递导致的重复执行

所以 MQ 解决了任务投递和消费扩展问题，但它没有消除业务幂等问题。只要消息可能重复投递，消费者就必须能承受重复执行。

6、数据库任务表和 MQ 应该如何选择

到这里，选择数据库任务表还是 MQ，就不能只看“哪个更强”，而要看当前系统的主要矛盾是什么。

如果文件解析任务量不大，但业务非常关心任务状态，例如要知道每个文件是否解析成功、失败原因是什么、能不能人工重新触发，那么数据库任务表更合适。它的优势是状态清晰、查询方便、补偿直接。

如果文件解析任务量很大，上传高峰明显，多个服务实例需要共同消费，而且系统更关心削峰和吞吐，那么 MQ 更合适。它的优势是投递能力强、消费者扩展方便、失败重投和死信机制更成熟。

可以用下面这个表收束选择条件：

判断问题 更倾向数据库任务表 更倾向 MQ
任务量是否很大 中小规模 高吞吐、大规模
是否需要频繁查询单个任务状态 需要 通常配合业务表或日志
是否需要多消费者扩展 可以做，但要自己抢占 天然支持消费组
失败重试和死信 自己实现 通常由 MQ 支持
接入成本 较低 中等，需要维护 MQ
主要目标 可恢复、可追踪、可补偿 削峰、解耦、高吞吐

最后一种组合在实际系统里也很常见。MQ 负责投递和削峰，业务任务表负责记录状态和支持查询。文件上传后，系统可以先写业务任务表，再发送 MQ 消息；Consumer 消费消息后，更新任务表状态。这样 MQ 负责“让任务被消费”，数据库负责“让任务可追踪”。

但组合方案也意味着复杂度上升。它需要处理发送消息和写数据库之间的一致性，需要设计幂等消费，也需要处理消息重复、消息延迟和死信补偿。因此，不要为了“架构完整”一开始就上组合方案，而要根据任务量、可靠性和排查需求逐步升级。

总结

本文可以从上一章的内存队列边界开始理解：因为内存队列只存在于当前 JVM，所以它能解决短暂排队和本机并发执行，却不能保证服务重启、机器宕机、多实例部署下任务仍然可靠。文件解析任务如果不能丢，就必须把任务从“内存中的临时对象”升级成“有稳定状态的任务记录”。

数据库任务表引入后，任务状态有了可恢复的位置，系统可以在重启后重新加载未完成任务，也可以通过状态抢占支持多实例处理。但数据库任务表更擅长状态管理和补偿，不适合无限承担高吞吐投递压力。当任务量继续增大，系统需要削峰、多消费者扩展和成熟的重投死信能力时，MQ 才成为更合适的投递通道。

最终，数据库任务表和 MQ 不是简单的替代关系。前者解决的是任务可恢复、可查询、可补偿，后者解决的是分布式投递、削峰和消费扩展。无论选择哪一种，只要任务可能被重复执行，业务幂等都不能省略；只要任务可能失败，重试和死信都必须有边界。

# 博客写作通用 Skill：问题驱动、角色建模与连贯推导

## 一、写作目标

文章不是知识点堆叠，而是帮助读者理解一个主题为什么出现、它由哪些关键角色组成、流程如何推进、关键结论为什么成立，以及它适合什么、不适合什么。

写作时默认读者对主题不熟。不要直接抛结论，也不要默认读者已经知道内部结构。核心概念第一次出现时，要讲清：

| 问题      | 要求            |
| ------- | ------------- |
| 它是什么    | 用简洁语言说明本质     |
| 它属于哪里   | 说明它在整体结构中的位置  |
| 它为什么需要  | 说明没有它会有什么问题   |
| 它如何参与流程 | 放回主线解释它推动了哪一步 |

后文再次使用同一个概念时，只做简短回指，不重复完整定义。

## 二、文章结构

文章应按“问题引入 → 角色关系 → 主例子 → 机制推导 → 对比和边界 → 总结”的思路组织，但不要机械套模板。章节顺序必须服务于理解：前一节要能自然引出后一节。

不建议把文章写成 API、名词或知识点罗列。章节标题要偏问题导向，例如：

```markdown
## 1. 为什么普通方案不够
## 2. 这个机制里有哪些关键角色
## 3. 主流程是如何一步步执行的
## 4. 为什么推荐这种写法
## 5. 它适合什么场景，又不适合什么场景
```

章节数量不要太多。能合并的内容尽量合并，尤其是：

| 情况                | 处理方式          |
| ----------------- | ------------- |
| 两节解释同一个机制的前后半段    | 合并            |
| 一节只是另一节的铺垫        | 合并            |
| 多节都在讲同一个例子的不同表层行为 | 合并成一个选择规则或对比表 |
| 后文重复前文定义          | 删除或改成回指       |

不应合并的情况是：两个章节抽象层级不同，或者读者必须先完成一个概念闭环，才能进入下一层。

## 三、开头摘要

Front Matter 后不要写一级标题 `#`。正文直接从摘要开始。

摘要要说明本节主要讲什么、为什么值得读，并自然引出后续内容。摘要不宜太长，通常 1 到 3 个自然段即可。

摘要不要写成目录复述，也不要一上来罗列概念。它应该告诉读者：本文要解决的核心问题是什么，以及读完后能理解什么。

示例：

```markdown
ForkJoinPool 不只是一个“特殊的线程池”。它真正要解决的问题，是当一个大任务不断拆成小任务时，如何让多个工作线程保持忙碌，并在任务分布不均时自动重新分配工作量。

本节会从任务提交、工作线程、本地队列和工作窃取几个角度，逐步解释 ForkJoinPool 如何执行分治任务，以及为什么它适合计算型任务，却不适合大量阻塞任务。
```

## 四、标题与层级格式

每篇文章开头必须有 YAML frontmatter，缩进两个空格，格式如下：

```markdown
---
title: 标题
date: 2026-07-05
tags:
  - 标签1
  - 标签2
categories:
  - 分类
---
```

frontmatter 之后**不再写一级标题（#）**，直接进入摘要段落。

摘要段落要做到：说清本节要讲什么问题、为什么值得读、和上一节的关系。摘要之后才开始正文分节。

分节统一用 `## 1、xxx`、`### 1.1 xxx` 的编号方式，章节顺序必须前后连贯（后一节的问题由前一节自然引出），**章节数量控制在 7～12 个之间**，能合并的小节尽量合并，不要为了"讲全"而拆得过细。

结尾单独写"总结"一节，不编号。

---

## 五、主例子使用规则

全文尽量使用一个主例子贯穿。主例子第一次出现时可以完整展开，后文复用时只做回指。

主例子要能承载文章主线，不要为了展示更多场景频繁更换例子。新增例子只有在旧例子无法解释新问题时才使用。

每段代码、案例或图示出现前后，都要说明它对应主线中的哪一步。不要只贴代码，也不要只展示结果。

代码解释时要讲清：

| 问题      | 要求               |
| ------- | ---------------- |
| 谁在执行    | 明确调用方或执行方        |
| 输入是什么   | 当前条件是什么          |
| 推进了什么   | 这一行或这一段让流程进入哪里   |
| 结果在哪里体现 | 返回值、状态变化或后续行为是什么 |

## 六、关键结论必须补推理

凡是出现“推荐”“更适合”“不适合”“最好不要”“核心区别”“正确做法”这类判断，都要解释原因。

不要只写：

```markdown
这种方式更适合复杂场景。
```

要写清：

```markdown
复杂场景中，输入和处理过程往往会分阶段变化。如果所有逻辑都集中在一个步骤里，后续调整会牵动整条流程。拆成职责明确的几个环节后，每个环节只处理自己的输入和输出，变化就更容易被限制在局部。
```

每个重要结论都尽量补足：

| 问题         | 要求          |
| ---------- | ----------- |
| 为什么不是另一种做法 | 说明被放弃方案的问题  |
| 为什么选择当前做法  | 说明当前做法解决了什么 |
| 当前做法有什么代价  | 说明边界和限制     |

## 七、图示规范

所有图示一律使用 Mermaid，禁止使用 ASCII 文本图（Unicode 框线等均不再使用）。Mermaid 图示适合表达角色关系、流程流转、状态变化、多个对象之间的协作关系。简单两步顺序关系不要画图，直接用文字说明。

### Mermaid 语法约束（Hexo Fluid 主题，Mermaid 8.14）

1. **标签中的 `.` 需要用引号包裹**：`A[Java Source .java]` 在 Mermaid 8.x 中会解析失败，应写成 `A["Java Source .java"]`。
2. **换行标签必须用 `<br>`，不能用 `<br/>`**：`obj["User @ 0x1000<br>age = 18"]` 正确，`<br/>` 自闭合形式在 8.14 中不兼容。
3. **含特殊字符（`=`、`@`、`.`）的标签统一用 `["..."]` 包裹**，避免解析歧义。
4. **标签文字尽量精简**：Mermaid 并排子图时，过长的标签会导致整张图横向溢出。每个节点的文字控制在 10 个字符以内，多余信息写到正文中。
5. **多个子图并排过宽时，优先缩短标签**，而不是改变布局方向。

图示只保留一处信息量最大的版本。后文复用相同结构时，用文字回指，不重复画图。

## 八、表格规范

Markdown 表格必须使用标准 Markdown 格式，不使用制表符。

概念较多、选择条件较多、多个方案需要横向比较时，优先用表格整理。表格适合用于：

| 场景        | 用途          |
| --------- | ----------- |
| 多个概念      | 对比含义、职责、位置  |
| 多种方案      | 对比适用场景和代价   |
| 多个阶段      | 对比输入、处理、输出  |
| 常见误区      | 对比错误理解和正确理解 |
| API 或方法选择 | 对比行为差异和使用场景 |

表格用于收束和对比，不替代正文推导。关键机制要先用自然段讲清，再用表格整理。

## 九、去重与合并

初稿完成后必须做一次检查：

| 检查项             | 处理方式          |
| --------------- | ------------- |
| 同一概念是否完整解释多次    | 保留首次解释，后文改成回指 |
| 同一例子是否重复展开      | 保留第一次完整展开     |
| 是否有结构相同的图       | 只保留信息量最大的一张   |
| 是否有章节只是换句话重复上一节 | 合并或删除         |
| 是否有和主线无关的旁支     | 删除、后移或拆成独立文章  |
| 总结是否逐节复述        | 改成因果链总结       |

判断一段内容是否该保留，可以看它是否帮助读者理解下一步为什么成立。如果不能，就应压缩或删除。

## 十、结尾总结

文章结尾必须有总结。

总结不要逐节复述，也不要重新列知识点。总结应该把全文串成一条更高层的因果链，回答：

| 问题           | 要求     |
| ------------ | ------ |
| 文章从什么问题开始    | 点出起点   |
| 为什么需要新结构或新方案 | 点出动因   |
| 中间关键角色如何协作   | 串起主线   |
| 最终解决了什么      | 回答文章标题 |
| 还有什么边界       | 防止过度泛化 |

总结可以使用自然段，不强制列表。它应该让读者在读完后形成整体理解，而不是再次看到正文的缩写版。

## 十一、最终交付要求

最终输出文章时，应满足：

| 要求                  |  是否必须 |
| ------------------- | ----: |
| 包含规范 Front Matter   |     是 |
| Front Matter 后先写摘要  |     是 |
| 不使用一级标题 `#`         |     是 |
| 章节使用 `## 1. ...`    |     是 |
| 子章节使用 `### 1.1 ...` | 需要时使用 |
| 章节数量不过多             |     是 |
| 能合并的章节尽量合并          |     是 |
| 主例子贯穿全文             |    优先 |
| 关键结论补足推理            |     是 |
| 必要时使用表格整理           |     是 |
| 文本图必须对齐             |     是 |
| 结尾有总结               |     是 |
| 不输出“全文核心概念清单”       |     是 |

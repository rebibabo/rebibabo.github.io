# 博客写作 Skill：角色建模 + 流程推导 + 边界收束

## 适用范围

用于撰写、润色、重写技术类博客文章：原理讲解、流程设计、方案对比、问题复盘等。

目标不是罗列知识点，而是让读者从"不知道这个东西为什么存在"，逐步走到"理解它的结构、流程和边界"。正文应是一条连续推导链，不是若干独立知识点的堆叠。

---

## 一、文章格式约定

每篇文章开头必须有 YAML frontmatter，缩进两个空格，格式如下：

```markdown
---
title: 标题
date: 2026-07-05
tags:
  - 标签1
  - 标签2
categories:
  - 分类
---
```

frontmatter 之后**不再写一级标题（#）**，直接进入摘要段落。

摘要段落要做到：说清本节要讲什么问题、为什么值得读、和上一节的关系。摘要之后才开始正文分节。

分节统一用 `## 1、xxx`、`### 1.1 xxx` 的编号方式，章节顺序必须前后连贯（后一节的问题由前一节自然引出），**章节数量控制在 7～12 个之间**，能合并的小节尽量合并，不要为了"讲全"而拆得过细。

结尾单独写"总结"一节，不编号。

---

## 二、先建立问题，再建立模型

写正文前，先想清楚：

1. 这个主题要解决什么问题？
2. 原来的做法为什么不够？
3. 新方案引入了哪些角色，各自的职责是什么？
4. 角色之间如何协作，事情如何一步步发生？

不要一上来就解释术语或堆结论。但也不要把推进方式钉死成单一直线（例如"问题→旧方案不足→新角色→关系建立→流程运转→机制解释→边界收束"这种固定七步）。不同主题的推导路径不同，允许两节合并、允许跳过某一步、允许把边界提前到中间讨论——只要整体读下来逻辑通顺即可。

核心要求只有一条：每一步都要让读者明白"为什么需要走到这一步"，而不是"又多了一个知识点"。

首次出现核心概念时，要说清楚：它是什么、属于谁、为什么需要它、它在流程中推动了哪一步。不要只写"A 可以提升效率"这种无因果的结论句。

---

## 三、角色关系与主例子

只要文章涉及多个对象、模块或参与方，就先用一两句话或一张简图交代角色关系（谁发起、谁接收、谁保存中间结果、谁决策、谁不能被替代），再进入细节。

全文尽量只用**一个主例子**贯穿，第一次完整展开，后文复用时只需一句回指（如"沿用前面的订单例子"），不必重新展开。只有当原例子确实承载不了新问题、且不破坏主线时，才引入简短的新例子。

---

## 四、关键结论必须补推理

出现"通常推荐……""更适合……""核心原因是……""正确做法是……"这类判断句时，必须补上因果：为什么不选 A，为什么选 B，B 解决了 A 的什么问题，B 又带来了什么新边界。禁止只写结论不写推理。

---

## 五、代码、图示、表格的使用规范

三者都是为了降低理解成本，不是装饰，使用前先确认：它对应正文哪一步、解决了什么"讲不清楚"的问题、看完后读者应得出什么结论。

**图示**：所有图示一律使用 Mermaid（Hexo Fluid 主题内置 Mermaid 8.14，无需额外插件），禁止使用 ASCII 文本图或 Unicode 框线图。Mermaid 语法约束：
- 标签中的 `.` 需要用引号包裹：`A["Java Source .java"]` 而非 `A[Java Source .java]`
- 换行用 `<br>` 不能用 `<br/>`
- 含 `=`、`@`、`.` 等特殊字符的标签统一用 `["..."]` 包裹
- **标签文字尽量精简**：多子图并排时标签过长会导致整张图横向溢出，每个节点控制在 10 字符以内，用 `<br>` 换行；仍过宽时用 `-->` 连线强制纵向排列
- **标签优先使用中文**：节点文字统一翻译为中文，与正文语言保持一致

**表格**：用于横向对比多个方案、多个角色职责、多个阶段输入输出，或概念较多需要速查时使用。表格放在正文推导之后做收束，不能替代推导本身，不能让读者直接看表格自己悟。

---

## 六、避免堆叠，保持层级一致

删除或压缩：和主线无关的背景、只出现一次不再复用的概念、没有新增信息的重复解释、为凑"完整性"硬加的旁支。判断标准只有一个：这段内容是否帮助读者理解下一步为什么成立，不能就删。

同一段落不要混合多个抽象层级（问题层、角色层、流程层、状态层、决策层、边界层、实现层）。先在当前层级讲完整，再往下一层走，不要在讲问题时突然扎进实现细节。

---

## 七、常用结构模板（三选一，按需增减小节）

| 模板 | 适用场景 | 骨架（4～6 节） |
|---|---|---|
| 概念解释 | 讲清一个新概念 | 问题由来 → 概念是什么及其位置 → 与相关角色的关系 → 一个例子跑通全流程 → 设计原因与适用边界 |
| 流程讲解 | 讲清一套运转机制 | 没有该流程的困境 → 角色模型 → 输入/转换/输出（可合并为一节分小节）→ 异常与边界 |
| 方案对比 | 比较多个方案 | 统一问题口径 → 方案 A 思路与边界 → 从 A 的不足引出 B → （如需）C 解决的新问题 → 选择条件 |

---

## 八、总结写法

总结不逐节复述，也不重新列知识点，而要写成一条因果链：文章从什么问题开始 → 为什么原方案不够 → 新角色/新流程如何引入 → 最终解决了什么 → 还留下什么边界没解决。

推荐句式："本文可以从……开始理解：因为……，所以需要……；当……出现后又带来……，于是引入……；最终这套方案解决的是……，但不负责……。"

---

## 九、交付前自检

| 检查项 | 要求 |
|---|---|
| frontmatter 格式与缩进 | 必须正确 |
| 是否误加一级标题 | 不允许 |
| 开头是否有摘要 | 必须有 |
| 章节编号是否为 ## 1、### 1.1 | 必须是 |
| 章节数量 | 控制在 4～7 个，能合并则合并 |
| 是否套用固定七步直线结构 | 不允许，按主题灵活组织 |
| 核心概念首次出现是否讲清"为什么在这里" | 必须有 |
| 是否只用一个主例子贯穿 | 优先满足 |
| 关键判断句是否补了推理 | 必须满足 |
| ASCII 图是否对齐、中文是否凑整 3 的倍数 | 必须检查 |
| 表格是否替代了推导 | 不允许，表格用于收束 |
| 是否存在知识点堆叠或层级跳跃 | 删除或调整 |
| 总结是否写成因果链 | 必须是 |
| 生成文件时是否提供下载链接 | 必须有 |

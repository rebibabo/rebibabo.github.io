# 可靠任务系统

[[wiki/concepts/concurrency/任务系统|返回任务系统与系统边界]]

## 定义

这一页关注的是：

> 当任务不能只放在内存里时，怎样让系统具备更强的可恢复性、可重试性和可靠交付能力。

单机内存队列能解决短暂排队和本机并发执行，但不能保证服务重启、机器宕机、多实例部署下任务仍然可靠。任务只存在于 JVM 内存里——重启后消失，多实例无法共享。

## 它解决什么问题

它会把几个系统级问题拉进来：

- 任务是否持久化
- 失败之后怎么恢复
- 数据库任务表和 MQ 该怎么取舍
- 幂等、重试、补偿怎么设计

## 数据库任务表：让任务可恢复

### 核心变化

数据库任务表不是简单地把 `BlockingQueue` 换成数据库，而是让数据库成为任务的**事实来源（source of truth）**。内存队列仍然可以存在，但只负责加速本机 Worker 取任务；真正判断任务有没有创建、有没有成功、是否需要重试，看数据库任务表。

提交任务必须先落库再入队列：

```java
taskRepository.insert(task);       // 先落库
taskQueue.offer(task);              // 再入队列
```

即使 `offer()` 失败（队列满了），任务也不算丢失——已写入数据库，后续由定时扫描器重新加载。

### 任务状态流转

```
CREATED → RUNNING → SUCCESS
CREATED → RUNNING → FAILED → RETRYING → RUNNING → SUCCESS / DEAD
```

重启后恢复：

| 数据库状态 | 重启后的处理 |
| --- | --- |
| `CREATED` | 重新加载到队列 |
| `RETRYING` | 到达重试时间后重新加载 |
| `RUNNING` | 如果更新时间过久，可以认为上次执行中断 |
| `SUCCESS` | 不再处理 |
| `DEAD` | 留给人排查或补偿 |

### 多实例防重复：状态抢占

```sql
UPDATE task_record
SET status = 'RUNNING', update_time = NOW()
WHERE id = ? AND status IN ('CREATED', 'RETRYING');
```

影响行数 = 1 → 抢到任务；影响行数 = 0 → 已被其他实例抢走。状态抢占尽量避免了多实例同时执行同一任务，但不能完全保证只执行一次（Worker 抢到任务后宕机）。因此还需要业务幂等兜底。

## 数据库任务表和 MQ 如何选择

| 判断问题 | 更倾向数据库任务表 | 更倾向 MQ |
| --- | --- | --- |
| 任务量 | 中小规模 | 高吞吐、大规模 |
| 是否需要频繁查询单个任务状态 | 需要 | 通常配合业务表或日志 |
| 是否需要多消费者扩展 | 可以做（自己抢占） | 天然支持消费组 |
| 失败重试和死信 | 自己实现 | 通常由 MQ 支持 |
| 接入成本 | 较低 | 中等，需要维护 MQ |
| 主要目标 | 可恢复、可追踪、可补偿 | 削峰、解耦、高吞吐 |

简单决策规则：

```
need clear task state? → use task table
need high throughput delivery? → use MQ
need both? → MQ + business task table
```

## MQ 的关键概念

| 概念 | 在流程中的作用 |
| --- | --- |
| Broker | 保存并投递消息 |
| Consumer | 接收消息并执行业务 |
| Ack | 告诉 Broker 消息已完成 |
| Retry | 临时失败后重新投递 |
| Dead Letter | 多次失败后隔离 |
| Idempotency | 兜底重复投递导致的重复执行 |

**Ack 的正确顺序**：先处理业务 → 业务成功后再 ack。不要先 ack 再处理（把可靠性窗口提前关闭）。

MQ 语义通常是"至少一次投递"——消息可能重复到达，因此幂等不能省略。

## 幂等设计

同一个任务即使被执行多次，最终业务结果也应和执行一次一致。实现方式：用 `task_id` 或 `biz_id` 作为幂等依据，处理前先检查是否已处理过。

任务表负责减少重复执行（状态抢占），幂等设计负责兜底重复执行。二者解决的问题不同，缺一不可。

## 为什么它重要

到这一步，并发已经不只是 JVM 内线程协作，而是任务生命周期管理。这也是"单机并发知识"通向"工程系统设计"的关键出口。数据库任务表和 MQ 不是替代关系——前者解决可恢复、可查询、可补偿，后者解决分布式投递、削峰和消费扩展。

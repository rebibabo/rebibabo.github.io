# ReentrantReadWriteLock

[[wiki/concepts/concurrency/并发容器|返回并发容器与数据结构]]

## 定义

`ReentrantReadWriteLock` 是把"读可以共享、写必须独占"这套语义封装成同步器的读写锁。

## 它解决什么问题

它回答的是：

> 如果读很多、写较少，能不能不要像互斥锁那样把所有读线程也互相堵住？

普通互斥锁无论读还是写都互斥。但从语义上看，多个线程同时读并不会破坏数据一致性，真正需要互斥的是读和写、写和写。

| 操作关系 | 是否可以并发 | 原因 |
| --- | --- | --- |
| 读 + 读 | 可以 | 多个线程只读取，不修改共享状态 |
| 读 + 写 | 不可以 | 写线程可能改变读线程正在读取的数据 |
| 写 + 写 | 不可以 | 多个写线程同时修改会破坏一致性 |

## 为什么它重要

它把锁语义进一步细分了：

- 读 + 读：可以并发
- 读 + 写：不可以
- 写 + 写：不可以

这让它成为"读多写少"场景里非常典型的同步模型。

## 关键设计

### 两个锁入口共享同一份状态

`ReadLock` 和 `WriteLock` 是两个入口，但背后是同一个 `Sync(AQS)`。读锁和写锁必须互相感知——读线程申请读锁时需要知道有没有写线程正在写，写线程申请写锁时需要知道有没有读线程正在读。

### state 拆分：高 16 位读锁，低 16 位写锁

AQS 的 `int state` 被拆成两部分：

```
state = 高 16 bit（读锁总数）+ 低 16 bit（写锁重入次数）
```

- 写锁增加一次：`state + 1`
- 读锁增加一次：`state + (1 << 16)`
- 读锁计数 = `state >>> 16`
- 写锁计数 = `state & 0xFFFF`

写锁放在低 16 位，因为和普通独占锁类似直接加减。读锁放在高 16 位，与写锁计数隔离。

### 读锁和写锁的申请规则

写锁（独占）：要求当前没有读锁；如果已经有写锁，只能是当前线程重入。

读锁（共享）：只排斥"别人持有的写锁"。如果写锁正由当前线程自己持有，当前线程也可以继续获取读锁。

压缩为两句话：**写锁怕任何读锁，也怕别人持有的写锁；读锁只怕别人持有的写锁。**

### 写锁可降级为读锁

当前线程持有写锁时可以继续获取读锁，从而支持锁降级——先获取读锁，再释放写锁，平滑从"独占写"过渡到"共享读"。关键是**先获取读锁，再释放写锁**——如果先释放写锁再去获取读锁，中间会出现无锁空档。

反过来锁升级通常不允许：持有读锁时申请写锁，写锁要求所有读锁释放，多个读线程都尝试升级就容易互相等待。

### 每个线程自己的读锁持有计数

`state` 高 16 位只记录读锁总数，不记录每个线程各自持有几次。因此还需要线程本地计数来保证释放时"只能减少自己的那一部分"。源码中通过 `firstReader`、`cachedHoldCounter` 等优化字段让常见路径更快。

### 读写线程共享同一条 AQS 队列

读锁失败后进入共享模式节点，写锁失败后进入独占模式节点。队列推进规则：连续读节点可以共享传播，遇到第一个写节点形成边界——避免写节点被不断越过而导致饥饿。

公平性影响插队行为。公平锁更严格尊重队列顺序；非公平锁更倾向于先尝试当前状态是否允许获取，但即使非公平读锁也会对"队列前方已有写节点等待"做一定限制，降低写线程饥饿风险。

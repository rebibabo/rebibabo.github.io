# 线程中断与安全停止

[[wiki/concepts/concurrency/故障模式|返回故障模式与安全停止]]

## 定义

这一页关注的是：

> 想让任务停下来时，怎样给线程一个"请停止"的信号，而不是粗暴终止它。

中断的核心不是"杀死线程"，而是"通知线程该停止了"。线程是否退出、什么时候退出、退出前是否清理资源，都取决于任务代码本身是否写好了响应路径。

## 它解决什么问题

它解释的是：

- 中断标记是什么
- 阻塞方法为什么会响应中断
- 为什么"请求停止"和"立刻杀死线程"不是一回事

## 为什么它重要

线程停止几乎总是工程问题，而不是单纯 API 问题。如果停止处理不好，就容易出现：资源没释放、半处理状态、线程池里任务残留。

## 中断机制的核心

### `interrupt()` 只是发出停止请求

调用 `worker.interrupt()` 只是把目标线程的中断标记设为 `true`，表示"有人希望你停下来"——线程不会因为这个标记自动退出。中断是一种**协作式停止机制**。

### 运行中的任务要主动检查

```java
while (!Thread.currentThread().isInterrupted()) {
    doWork();
}
cleanup();
```

### 阻塞中的任务通过 InterruptedException 被唤醒

线程阻塞在 `BlockingQueue.take()` 等可中断方法时，外部 `interrupt()` 会让它抛出 `InterruptedException` 从而醒来。

### catch 里不能吞掉中断

阻塞方法抛出 `InterruptedException` 后通常会清除中断标记。所以 catch 里要恢复：

```java
catch (InterruptedException e) {
    Thread.currentThread().interrupt(); // 恢复中断标记
    return;                              // 真正退出
}
```

### `isInterrupted()` vs `Thread.interrupted()`

| 方法 | 作用对象 | 是否清除中断标记 |
| --- | --- | --- |
| `thread.isInterrupted()` | 指定线程 | 否 |
| `Thread.interrupted()` | 当前线程 | 是 |

任务循环中用 `isInterrupted()`，避免标记被意外清除。

## 一个完整的可停止 Worker

```java
class StoppableWorker implements Runnable {
    @Override
    public void run() {
        try {
            while (!Thread.currentThread().isInterrupted()) {
                Task task = queue.take();       // 阻塞中响应中断
                process(task);
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt(); // 恢复中断标记
            return;                              // 退出
        } finally {
            cleanup();                           // 清理资源
        }
    }
}
```

三个部分缺一不可：运行中检查中断、阻塞中响应中断、退出前清理资源。

## 线程池关闭

```java
executor.shutdown();                              // 不再接收新任务
if (!executor.awaitTermination(30, SECONDS)) {
    executor.shutdownNow();                       // 超时升级，尝试中断 Worker
}
```

| 方法 | 是否接收新任务 | 队列中未开始任务 | 正在执行的任务 |
| --- | --- | --- | --- |
| `shutdown()` | 否 | 继续执行 | 继续执行 |
| `shutdownNow()` | 否 | 从队列移除并返回 | 尝试中断 |

`shutdownNow()` 返回的是队列里还没开始执行的任务，不包含正在执行中的任务——它们只会收到中断请求。

## 哪些阻塞能被 interrupt 唤醒

| 方法 | 中断后的表现 |
| --- | --- |
| `Thread.sleep()` | 抛 `InterruptedException` |
| `Object.wait()` | 抛 `InterruptedException` |
| `BlockingQueue.take()` | 抛 `InterruptedException` |
| `Condition.await()` | 抛 `InterruptedException` |
| `ReentrantLock.lockInterruptibly()` | 抛 `InterruptedException` |

**注意：`synchronized` 等锁不能被 `interrupt()` 唤醒。** 如果希望等锁时也能响应中断，用 `ReentrantLock.lockInterruptibly()`。

## `volatile running` 和 `interrupt()` 的选择

| 场景 | 推荐方式 | 原因 |
| --- | --- | --- |
| 简单非阻塞循环 | `volatile running` 或 `interrupt()` | 线程能主动检查 |
| 可能进入 `sleep/wait/take` | `interrupt()` | 可以唤醒可中断阻塞 |
| 等待 `synchronized` 锁 | `interrupt()` 不够 | 只能设置标记，不能唤醒锁等待 |

## 为什么不用 `Thread.stop()`

`Thread.stop()` 强制终止线程，可能停在任意位置——对象处于半完成状态，锁被释放后其他线程看到不一致状态。`interrupt()` 不会这样做：线程仍然可以完成当前临界区、执行 `finally`、释放资源，然后在自己可控的位置退出。

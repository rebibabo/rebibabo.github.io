# 缓存一致性

[[wiki/concepts/concurrency/执行模型|返回执行模型]]

## 定义

缓存一致性关注的是：

> 多核 CPU 各自都有缓存时，同一个共享数据的不同副本怎样保持协调？

## 它解决什么问题

它解释的是硬件层面的一个关键背景：

- 一个核心刚写完数据
- 另一个核心为什么不一定立刻就看到
- 为什么同一条 cache line 会在不同核心之间转移、失效、再生效

## 同一份数据为什么会有多个副本

从 Java 代码看，一个字段只有一份；但从硬件执行过程看，它所在的数据可能同时出现在多个 Core 的 Cache 中。

这并不矛盾，因为 CPU 为了减少每次都访问主内存的成本，会把最近使用的数据缓存到离自己更近的位置。

所以当两个线程分别跑在不同 Core 上时，同一个共享字段可能出现这样的情况：

- Core 1 的 Cache 里有一份副本
- Core 2 的 Cache 里也有一份副本
- 主内存里还有“最终来源”的那份数据

一旦某个 Core 修改了自己的副本，就必须回答一个问题：

> 其他 Core 什么时候知道自己的旧副本已经过期？

这就是缓存一致性要解决的核心。

## CPU 不是按字段，而是按 Cache Line 管理

硬件通常不会只缓存某一个 Java 字段，而是按一整条 `Cache Line` 读写和维护状态。

这意味着两件事：

1. 你改的也许只是一个 `int`，但硬件协调的是整条缓存线
2. 两个不同字段只要落在同一条缓存线上，也可能互相影响

这也是后面理解伪共享、CAS 热点竞争的基础。

## 写入不等于立刻写回主内存

很多人第一次理解可见性时，会把“线程写了变量”直接想成“主内存立刻变了”。

实际更常见的情况是：

- Core 先修改自己的缓存副本
- 该副本被标记为更新过
- 之后再由硬件决定何时写回更低层缓存或主内存

所以另一个线程能否看到最新值，关键不只是“是否回写主内存”，而是：

- 旧副本有没有失效
- 下次读取时能不能拿到更新后的那份数据

## MESI 可以先当成一个理解模型

原文里最重要的不是死记状态名，而是理解这个思路：

- 多个 Core 可以共享只读副本
- 一旦某个 Core 要写，就必须先拿到独占修改权限
- 其他 Core 的旧副本会变成无效

也就是说，硬件层必须保证：

> 同一条 Cache Line 不能被多个 Core 同时以“可修改状态”持有。

这个“独占写入权”是硬件意义上的，不是 Java 锁。

## 它为什么会影响 CAS

CAS 看起来是“无锁”的，但它并不是“没有竞争”。

如果很多线程同时更新同一个原子变量，对应的 Cache Line 修改权限会在多个 Core 之间来回转移：

- 某个线程 CAS 成功
- 其他线程 CAS 失败后重试
- 同一条缓存线反复失效、重新获取

所以 CAS 避免的是线程阻塞，不是底层缓存层面的争抢。

## 它为什么还不等于 Java 可见性

缓存一致性解决的是：

> 同一个内存位置的多个缓存副本，怎样最终保持协调。

但 Java 并发还关心另外两件事：

- 多个变量之间要按什么顺序对外可见
- 编译器和 CPU 哪些重排结果是允许的

所以只有 `Cache Coherence` 还不够，还需要 [[wiki/concepts/concurrency/volatile|volatile]] 和 [[wiki/concepts/concurrency/Java内存模型|Java Memory Model]] 在语言层再建立规则。

## 为什么它重要

它不是 Java 并发语义本身，但它解释了很多现象的底层土壤：

- 可见性为什么不是“理所当然”
- CAS 高竞争时为什么会出现热点争抢
- 为什么 `volatile` / `JMM` 需要在语言层再建规则

## 在系列里的位置

它位于执行模型和 JMM 之间，是“硬件行为如何影响并发语义”的过渡节点。

它回答的是：

- 为什么共享变量会出现多个缓存副本
- 为什么写入传播不是零成本
- 为什么“最终一致”不等于“线程立刻可见”

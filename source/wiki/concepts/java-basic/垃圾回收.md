# 垃圾回收

[[wiki/concepts/java-basic/JVM|返回 JVM]]

## 这一层回答什么问题

> JVM 怎么判断一个对象可以回收了？新生代和老年代的回收策略为什么不同？从 Serial 到 ZGC，垃圾回收器到底在优化什么？

垃圾回收是 Java 内存管理的核心——你不写 `free()`，但 JVM 在背后做的大量决策会直接影响你的服务响应时间。

## 怎么判断"垃圾"

**引用计数法**（Java 不用）：记录每个对象的被引用次数。问题是解决不了循环引用——A 引用 B，B 引用 A，计数永远不为 0。

**可达性分析**（Java 实际使用）：从 **GC Roots** 出发，沿着引用链能到达的对象都是活的，到不了的就是垃圾。

GC Roots 包括：栈中引用的对象、静态变量引用的对象、常量引用的对象、JNI 引用的对象。

## 四种引用

| 类型 | 回收时机 | 用途 |
|------|----------|------|
| 强引用 | 不可达才回收 | 普通 `Object o = new Object()` |
| 软引用 | OOM 前回收 | 缓存（内存紧张时释放） |
| 弱引用 | 下次 GC 就回收 | `WeakHashMap` |
| 虚引用 | 随时回收 | 跟踪对象回收时机 |

## 三种算法

| 算法 | 怎么做 | 优点 | 代价 |
|------|--------|------|------|
| 标记-清除 | 标记垃圾 → 清除 | 简单 | 内存碎片 |
| 标记-复制 | 存活对象复制到空区 → 清空原区 | 无碎片 | 浪费一半空间 |
| 标记-整理 | 存活对象移到一端 → 清边界外 | 无碎片、不浪费 | 耗时 |

**新生代用标记-复制**（对象大部分朝生夕死，复制活下来的很少）。**老年代用标记-清除或标记-整理**（存活率高，不适合复制）。

## 回收器演进

每一次换代都在降低 **STW（Stop The World）的时间**——所有用户线程暂停、JVM 专心做 GC 的时间窗口。

| 回收器 | 年代 | 核心思路 | STW |
|--------|------|----------|-----|
| Serial | 最古老 | 单线程 | 长 |
| Parallel | Java 5-8 | 多线程并行 | 较长 |
| CMS | Java 5-9 | 并发标记、低延迟 | 较短 |
| **G1** | Java 9+ 默认 | 分区 Region、可预测暂停 | 短且可控 |
| ZGC | Java 15+ | 染色指针 | <1ms |

**G1** 把堆分成大小相等的 Region，优先回收垃圾最多的 Region（Garbage First）。**ZGC** 通过染色指针在引用里嵌入状态信息，实现亚毫秒级暂停——TB 级堆也在 1ms 以内。

## Minor GC vs Major GC vs Full GC

- **Minor GC**：只清理新生代，频繁但快
- **Major GC**：清理老年代，慢
- **Full GC**：清理整个堆 + 元空间，最慢——**线上要尽量避免**

## 排查工具

`jps` 找 PID → `jstat -gc <pid> 1000` 观察 GC 频率和时间 → `jmap -dump` 导出堆快照 → MAT / Arthas 分析。

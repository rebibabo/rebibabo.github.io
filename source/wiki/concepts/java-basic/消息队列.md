# 消息队列

[[wiki/concepts/java-basic/工程实践|返回工程实践]]

## 这一层回答什么问题

> 消息队列解决了什么？RabbitMQ 和 Kafka 本质区别是什么？消息丢了怎么办？重复消费怎么处理？

MQ 的核心价值是三件事：**解耦**（A 发消息、B 订阅，互不知道对方存在）、**异步**（非核心逻辑不在主流程里执行）、**削峰**（MQ 缓冲，消费者匀速消费，保护 DB）。

## 核心概念

```
Producer → Broker → Consumer
            ↕
          Queue / Topic
```

Producer 发消息到 Broker，Consumer 从 Broker 拉消息。两者不直接通信——任何一方挂了，另一方不受影响。

## RabbitMQ vs Kafka

| | RabbitMQ | Kafka |
|------|----------|-------|
| 设计 | 消息代理（AMQP 协议） | 分布式日志 |
| 吞吐 | 万级/秒 | 百万级/秒 |
| 消息删除 | ACK 后删除 | 消费后不删除（按时间/大小清理） |
| 核心概念 | Exchange + Queue | Topic + Partition |
| 适用 | 业务消息、任务分发 | 流数据、事件溯源、日志 |

**Kafka 为什么快**：顺序写磁盘（追加日志）、Page Cache + 零拷贝（数据从内核到网卡不经过用户空间）、批量发送压缩。

## 消息可靠性：三环节防丢失

| 环节 | 可能丢在哪 | 解决方法 |
|------|-----------|----------|
| 生产 → Broker | 网络、Broker 宕机 | Publisher Confirm、`acks=all` |
| Broker 存储 | 磁盘故障 | 多副本、`min.insync.replicas` |
| Broker → 消费 | 消费者挂了 | 手动 ACK（处理完再确认） |

## 重复消费：幂等性

"至少一次"语义下，同一消息可能被消费多次。消费者必须保证幂等：
- 唯一索引（业务 ID 写 DB，唯一约束兜底）
- Redis `SETNX` 去重（消息 ID 做 key）

## 顺序性

Kafka 同一 Partition 内消息有序。同一业务实体的消息发到同一个 Partition（通过 key hash），消费者单线程处理。

## 前言

前面两篇——[指标](/2026/05/05/Java-advanced/05-metric/) 和 [Grafana 大盘](/2026/05/06/Java-advanced/06-grafana/)——已经把可观测性三大支柱里的 **Metrics（指标）** 讲完了，还把它画成了能实时盯的大盘。指标能告诉你"系统整体健康吗"——QPS 多少、成功率多少、P99 耗时多少。但当大盘告诉你"下单接口变慢了"，你接着要问的是：**到底慢在哪一步？** 这个请求在我自己服务里慢，还是卡在调用支付接口上，还是数据库查询拖了后腿？

指标回答不了这个问题，因为它是"聚合后的数字"，丢掉了单次请求的细节。这就轮到第三根支柱 **Tracing（链路追踪）** 登场了。这篇不堆代码，重点把"一个请求是怎么被全程跟拍下来的"这件事的原理讲透——它其实没那么玄，核心思想用一个快递单号就能解释清楚。

## 1. 先感受痛点：微服务把"排查"变难了

### 1.1 单体时代，排查很简单

回到只有一个应用、一个数据库的时代。一个请求进来，从 Controller 到 Service 到 Mapper，全在**同一个进程、同一个线程**里跑完。出了问题怎么办？

- 看日志：所有日志都在一个文件里，按时间顺序往下翻就行
- 看耗时：在方法前后打两个时间戳，相减就知道哪段慢
- 看堆栈：报错了直接有完整的异常堆栈，一眼看到是哪行代码

本质原因是：**所有事情都发生在一个地方，天然就是连贯的。**

### 1.2 微服务时代，请求被"打散"了

现在一个"用户下单"的请求，可能要经过这样一条链路：

```
用户 → API 网关 → 订单服务 → 库存服务 → 支付服务 → 数据库
                      │
                      └→ 优惠券服务
```

同一个请求，被拆到了 5、6 个**不同的进程**里执行，这些进程可能还跑在不同的机器上。这时候原来的排查手段全部失灵了：

| 单体里好用的手段 | 到了微服务里 |
|---|---|
| 翻一个日志文件 | 日志分散在 6 台机器、6 个文件里，怎么知道哪几条属于同一个请求？ |
| 方法前后打时间戳 | 每个服务只知道自己花了多久，谁也不知道"整条链路"的全貌 |
| 完整异常堆栈 | 支付服务报错了，订单服务只收到一个"调用失败"，看不到根因 |

一个很形象的说法：单体时代你在**追一个人**，他从头到尾在你眼前；微服务时代你在**追一个在六个房间之间穿梭的人**，每个房间的监控只拍到他在本房间的片段，你需要一种办法把这些片段**按同一个人、按先后顺序拼起来**。

链路追踪要解决的，就是这件"把碎片拼回完整故事"的事。

## 2. 核心思想：给每个请求发一张"快递单号"

链路追踪的整个体系，建立在一个朴素到不能再朴素的想法上：

> **给每一个进入系统的请求，发一个全局唯一的编号；这个编号跟着请求一路传下去，每个服务做任何事都带上它。**

这个编号就叫 **TraceId**（追踪 ID）。

想象你网购寄一个快递。快递单号 `SF123456` 从你下单那一刻生成，之后无论包裹到了哪个中转站、上了哪辆车、由哪个快递员派送，**每一条流转记录都带着这个单号**。最后你在 App 里输入单号，就能看到一条完整的物流时间线：

```
SF123456
  10:00  已揽收（深圳）
  14:30  到达中转中心（深圳）
  22:00  已发出（深圳 → 北京）
  次日 08:00  到达派送点（北京）
  次日 11:00  已签收
```

链路追踪干的就是一模一样的事——只不过把"快递单号"换成 TraceId，把"中转站、快递车"换成"网关、订单服务、支付服务"。**TraceId 就是把分散在各处的记录串成一条线的那根线。**

但只有一个 TraceId 还不够。物流时间线里每一行是一个"环节"，链路追踪里也需要把一次请求里的每一段操作单独记下来——这就引出了第二个核心概念：**Span**。

---

## 3. 两个核心概念：Trace 与 Span

这是整个链路追踪里最需要理解清楚的两个词，理解了它俩，后面全是水到渠成。

### 3.1 Span：一段"有起止时间的工作"

**Span（跨度）** 表示一次操作、一段工作。它最核心的信息就是：**做了什么 + 从什么时候开始 + 到什么时候结束**。

比如这些都可以是一个 Span：

- 网关处理这个请求（从收到到转发出去）
- 订单服务执行下单逻辑
- 订单服务调用支付服务的这次远程调用
- 一次数据库查询

每个 Span 至少包含这些字段：

| 字段 | 含义 | 类比快递 |
|---|---|---|
| `spanId` | 这一段操作自己的唯一 ID | 这一条流转记录的行号 |
| `traceId` | 它属于哪一次完整请求 | 快递单号（同一单的所有行都一样） |
| `parentSpanId` | 它是被谁触发的（上一段是谁） | "我是从哪个环节流转过来的" |
| 操作名 | 做了什么，如 `POST /order` | "到达中转中心" |
| 开始 / 结束时间 | 这段花了多久 | 10:00 揽收、14:30 到站 |
| tags / 属性 | 附加信息，如状态码、SQL 语句 | 备注 |

### 3.2 Trace：一堆 Span 组成的"一棵树"

**Trace（一条追踪）** = 一次完整请求里产生的**所有 Span 的集合**。它们靠 `traceId` 相同绑在一起，又靠 `parentSpanId` 形成**父子关系**，最终拼成一棵树（或者说一张调用关系图）。

还是用下单的例子，一次请求产生的 Span 树长这样：

```
Trace: traceId = abc123
│
└─ Span A  网关处理请求            [0ms ────────────────── 850ms]
   └─ Span B  订单服务下单          [20ms ──────────────── 840ms]
      ├─ Span C  调用库存服务        [50ms ──── 180ms]
      ├─ Span D  调用优惠券服务      [60ms ── 120ms]
      └─ Span E  调用支付服务        [200ms ─────────── 800ms]  ← 慢！
         └─ Span F  支付服务查数据库  [250ms ──── 780ms]  ← 根因在这
```

- 所有 Span 的 `traceId` 都是 `abc123`（同一次请求）
- Span B 的 `parentSpanId` 指向 Span A，C/D/E 的 parent 指向 B……父子关系就是这么连起来的
- 每个 Span 自带起止时间，画在时间轴上就成了上面这种"甘特图"

**看懂这张图，你一眼就能得出结论**：整条链路 850ms，其中调用支付服务花了 600ms（200→800），而支付服务里又有 530ms 耗在数据库查询上。慢的根因，定位到了。

这正是链路追踪最大的价值：**它不只告诉你"慢"，还把"慢在哪、为什么慢"用一张图摆在你面前。**

> 小结一句话区分：**Trace 是整次请求，Span 是其中的一步；一个 Trace 由多个 Span 按父子关系组成。**

## 4. 最关键的原理：上下文是怎么"跨服务传递"的

到这里你可能有个疑问：道理我懂了，但**订单服务怎么知道自己的 traceId 是 `abc123`？它又怎么把这个 traceId 告诉下游的支付服务？** 这是整个链路追踪最核心、也最容易被忽略的机制，叫 **上下文传播（Context Propagation）**。

它要解决两个层面的传递：**一个进程内部**怎么传，**跨进程/跨服务**怎么传。

### 4.1 进程内传递：靠 ThreadLocal "悄悄带着走"

在**同一个服务内部**，一个请求通常由**同一个线程**从头处理到尾（Controller → Service → Mapper）。那这个请求的 traceId、当前 spanId 存哪？

答案是存在 **ThreadLocal** 里——这是 Java 里"线程私有变量"的机制，可以理解成"每个线程身上挂着的一个小口袋"。请求一进来，框架就把当前的追踪上下文（traceId、当前 spanId）塞进这个线程的小口袋里。之后这个线程跑到任何一层代码，想用的时候伸手到口袋里一掏就有了，**不需要在每个方法的参数里一层层传 traceId**。

```
请求进入 → 线程 T1 的口袋里放入 {traceId=abc123, spanId=A}
   ↓ T1 一路执行
Controller 想打日志 → 从口袋掏出 traceId=abc123
Service 要建子 Span → 从口袋掏出当前 spanId=A 作为新 Span 的 parent
   ↓
请求结束 → 清空 T1 的口袋（很重要，否则会"串味"影响下一个请求）
```

这也解释了一个常见的坑（第 8 节细讲）：一旦你把活儿丢到**线程池/异步线程**里执行，新线程是另一个口袋，里面是空的，上下文就"丢"了。

### 4.2 跨进程传递：把上下文塞进 HTTP 请求头

进程内靠口袋，可一旦订单服务要通过 HTTP **调用支付服务**，那就是两个进程、两个线程、两个完全独立的"口袋系统"了。口袋里的东西没法隔着网络自动跑过去。

办法很直接：**调用方在发出 HTTP 请求时，把追踪上下文写进请求头（Header）里带过去；被调用方收到请求时，再从请求头里把它读出来，放进自己的口袋。**

这就像快递包裹从深圳的车转到北京的车，工作人员会把"单号 + 来源"这张小标签**贴在包裹外面**一起交接，新一站撕下来登记，继续往下贴。

具体传什么？业界已经有了统一标准 **W3C Trace Context**，它规定用一个叫 `traceparent` 的请求头来携带：

```
请求头里多了这么一行：
traceparent: 00-abc123def456...-00f067aa0ba902b7-01
             │  │                │                │
            版本  traceId         parentSpanId    采样标志
```

- 调用方（订单服务）发请求前，把"我的 traceId + 我当前的 spanId"编码进 `traceparent` 头
- 被调用方（支付服务）一收到请求，解析这个头，就知道了："哦，我属于 traceId=abc123 这条链路，我的上游 span 是那个,我要新建一个 span 并把 parent 指向它"

整个传递链路串起来就是：

```
网关        生成 traceId=abc123, spanId=A
  │  HTTP 请求头带上 traceparent: ...abc123-A...
  ▼
订单服务    从头里读出 traceId=abc123, parent=A → 自己新建 spanId=B
  │  调用支付时，HTTP 头带上 traceparent: ...abc123-B...
  ▼
支付服务    从头里读出 traceId=abc123, parent=B → 自己新建 spanId=E
```

**注意全程 traceId 始终是 abc123 没变**（这是把所有 Span 绑在一起的关键），变的是每一跳新建自己的 spanId、并把 parent 指向上一跳。

> 划重点：链路追踪能"跨服务"，靠的不是什么黑魔法，就是**在每次远程调用的请求头里夹带 traceId 这张小标签**。换成 RPC、消息队列也一样——RPC 把它放进调用的元数据里，消息队列把它放进消息的 header 里，原理完全相同。

## 5. 数据是怎么被收集起来的：埋点与上报

上下文传播解决了"每个服务都知道自己属于哪条链路"。但这些 Span 数据产生之后，得有人**收集、汇总、存起来、画成图**，你才能看到第 3 节那张漂亮的树。这一节讲数据是怎么从代码里"冒出来"再"汇集"的。

### 5.1 埋点（Instrumentation）：Span 是在哪儿产生的

"埋点"就是在代码的关键位置插入"开始一个 Span / 结束一个 Span"的逻辑。有两种方式：

**① 手动埋点**——你自己在代码里写：

```java
Span span = tracer.spanBuilder("查询用户信用分").startSpan();
try {
    // ... 你的业务逻辑
} finally {
    span.end();   // 结束，记录耗时
}
```

灵活，但要一处处自己写，容易漏。一般只用来标记**业务上你特别关心的关键步骤**。

**② 自动埋点**——这才是主力。你**几乎不用改代码**，框架/探针会自动在这些地方帮你开始和结束 Span：

- 收到一个 HTTP 请求（Spring MVC 的拦截器层）
- 发起一次 HTTP/RPC 调用（HTTP 客户端层）
- 执行一条 SQL（JDBC 驱动层）
- 操作一次 Redis、发一条 MQ 消息

它是怎么做到"不改代码"的？常见手段是 **Java Agent + 字节码增强**：启动时加一个 `-javaagent` 参数，探针会在类加载的瞬间，往这些常用框架的方法里"悄悄织入"埋点代码（思路和你学过的 AOP 很像，只不过织得更底层）。所以你看到很多接入文档就一句话——启动命令加个 agent jar，链路就自动有了，原因就在这。

### 5.2 采样（Sampling）：不是每个请求都记

生产环境 QPS 可能几万几十万，如果**每一个请求**都完整记录 Span 再上报存储，数据量会大到存不下、也太贵。所以引入**采样**——只记录一部分请求。

最常见两种策略：

| 策略 | 怎么决定记不记 | 特点 |
|---|---|---|
| 头部采样（Head-based） | 请求**一进来**就掷个骰子决定，比如"只记 10%"，决定后整条链路统一遵守 | 简单、开销小，但可能恰好漏掉那个出错的请求 |
| 尾部采样（Tail-based） | 等整条链路**跑完**再决定，比如"出错的、慢的一定记，正常的只留一点" | 能保住有价值的链路，但要先把所有 Span 缓存起来，复杂、占内存 |

那个 `traceparent` 头最后一位（`01`/`00`）就是采样标志位：网关在头部采样时一旦决定"这条要记"，就把标志位设成 `01` 带下去，**保证一条链路要么整条都记、要么整条都不记**，不会出现半截的残链。

### 5.3 上报与汇总：Span 怎么从各服务汇到一处

每个服务把自己产生的 Span 先在内存里攒一小批（buffer），然后**异步**发给一个集中的收集器（Collector）。注意是异步——绝不能让"上报这件事"阻塞你的业务请求，这和上一篇讲指标时强调的"埋点不能拖慢主流程"是同一个道理。

收集器把来自各个服务、但 traceId 相同的 Span **按 traceId 归拢到一起**，根据 parentSpanId 重建出父子树，存进后端存储，最后由 UI 查询展示。

## 6. 整体架构：从产生到看见的全景图

把上面几节串起来，一条 Span 数据从"产生"到"被你看见"，要走完这条流水线：

```
①各个服务（埋点产生 Span）
   订单服务 ─┐
   支付服务 ─┤  异步上报（同一 traceId 的 Span 分散在各服务）
   库存服务 ─┘
        │
        ▼
②Collector 收集器
   按 traceId 把散落的 Span 归拢、重建成树
        │
        ▼
③存储（时序/检索数据库）
   存下所有 Trace，支持按 traceId、按耗时、按服务检索
        │
        ▼
④UI 可视化
   画成第 3 节那种甘特图，让你一眼看出慢在哪
```

这套流水线，市面上的主流工具基本都是这个骨架，只是分工和侧重不同：

| 工具 | 定位 | 一句话 |
|---|---|---|
| **OpenTelemetry（OTel）** | 标准 + 工具集 | 当下事实标准，定义了 API、数据格式、传播协议；负责"产生和传输" |
| **Jaeger** | 后端 + UI | Uber 开源，负责收集、存储、展示，常和 OTel 搭配 |
| **Zipkin** | 后端 + UI | Twitter 开源，元老，轻量 |
| **SkyWalking** | 一体化 APM | 国产，自带 Agent（自动埋点强）+ 存储 + UI，还含指标告警 |

**这里有个重要趋势要记住**：早期各家数据格式、传播协议互不相通，换工具就得重新接入。现在 **OpenTelemetry 把"怎么埋点、上下文怎么传、数据长什么样"统一成了标准**——你的应用只对接 OTel 这套标准 API，后端想用 Jaeger 还是别的，随时能换，业务代码不用动。这和指标那篇里 Micrometer 当"监控门面"是完全一样的思路：**用一层统一标准，把业务代码和具体的后端实现解耦。**

## 7. 在 Spring Boot 里怎么落地（少量代码）

原理懂了，落地其实很轻。现在 Spring Boot 生态推荐 **Micrometer Tracing**（前身是 Spring Cloud Sleuth）作为门面，底层对接 OpenTelemetry。

**第一步，加依赖**（门面 + 一个具体的上报实现，比如上报给 Zipkin/OTel）：

```xml
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-otel</artifactId>
</dependency>
<dependency>
    <groupId>io.opentelemetry</groupId>
    <artifactId>opentelemetry-exporter-zipkin</artifactId>
</dependency>
```

**第二步，配一下采样率和上报地址**：

```yaml
management:
  tracing:
    sampling:
      probability: 0.1        # 采样 10%，生产别设 1.0
  zipkin:
    tracing:
      endpoint: http://localhost:9411/api/v2/spans   # 上报到哪
```

就这样——**HTTP 请求、远程调用、数据库查询的 Span 都自动产生了**，traceparent 头自动在服务间传递，你基本不用写埋点代码。只有想标记某段特殊业务时，才手动补一个 Span。

**第三步，也是最实用的一步：让日志带上 traceId。** 这一步把"链路追踪"和你已有的"日志"打通了。在日志格式里加上 traceId/spanId（Micrometer Tracing 会自动放进日志的 MDC，你只要在 pattern 里引用）：

```
日志格式里加 [%X{traceId:-},%X{spanId:-}]，打出来就是：

[abc123,B] 2026-05-06 13:05:01  订单服务  开始处理下单
[abc123,E] 2026-05-06 13:05:01  支付服务  调用支付网关超时
```

这样一来，你在追踪 UI 上发现某条 trace 出错，**直接拿这个 traceId 去日志系统一搜，6 个服务里属于这次请求的日志全都跳出来了**。指标 → 链路 → 日志，三层就这么串通了。

## 8. 几个一定会踩的坑（深入原理）

这一节挑几个"原理上不讲清楚就一定会出问题"的点，理解了能少走很多弯路。

### 8.1 异步 / 线程池里，上下文会"丢"

回到第 4.1 节那个"线程口袋"。上下文存在**当前线程**的 ThreadLocal 里，一旦你这样写：

```java
threadPool.submit(() -> {
    callPaymentService();   // 这里 traceId 没了！
});
```

任务被丢进线程池，由**另一个线程**执行，那个线程的口袋是空的，traceId/spanId 全丢了。结果就是：异步那一段的 Span 要么挂不到原来的树上（变成一棵孤立的小树），要么干脆没记录，链路断了。

**原理上的解法**叫"上下文搬运"：在提交任务前，把当前线程口袋里的上下文**拷贝一份**，等任务在新线程开始执行时**先把它放进新线程的口袋**，结束再清掉。好在主流框架都提供了包装好的工具（比如包一层 `ContextSnapshot`、或用框架提供的 `TraceableExecutorService`），你只要记住一条原则：**凡是跨线程，就要主动把追踪上下文带过去。**

### 8.2 跨机器的"时钟不一致"

第 3 节那张甘特图，是把不同服务、不同机器上 Span 的开始/结束时间画在**同一根时间轴**上的。但不同机器的系统时钟并不完全一致（哪怕都装了 NTP 对时，也可能差几毫秒到几十毫秒）。

这会导致一些"反直觉"的画面：子 Span 的开始时间看起来比父 Span 还早、或者两段调用的先后顺序看起来不对。**这不是 bug，是物理上的时钟漂移。** 理解这一点后，看链路图时要明白：**跨服务的绝对时间点有少量误差是正常的，真正可靠的是每个 Span "自己机器上测出的持续时长"**。排查耗时优先看每段 Span 的 duration，而不是死抠绝对时间点的对齐。

### 8.3 采样率和"按需追踪"的矛盾

采样 10% 省了资源，但万一用户投诉的那次请求，恰好是没被采样的 90%，你就查无此 trace 了。这就是头部采样的天然缺陷。

实践中的折中思路：

- 对**出错/超时**的请求强制采样（尾部采样的思想，宁可贵点也要留住有问题的）
- 支持"**按 traceId 透传强制采样**"：从入口带一个强制标志，让某一类重点请求 100% 记录
- 普通正常流量低采样率，够看趋势就行

记住这是个**成本 vs 完整性**的权衡，没有银弹，按业务重要性调。

## 9. 三大支柱怎么配合：Metrics、Tracing、Logging

到这里，可观测性三件套你就齐了。它们不是三选一，而是**分工协作、层层下钻**：

```
①Metrics  指标大盘告警："下单接口 P99 从 200ms 涨到 2s 了！"
              │ 知道"有问题"，但不知道"为什么"
              ▼
②Tracing  翻出几条慢的 trace："原来都卡在调用支付服务那一段"
              │ 知道"卡在哪一步"，但不知道"那一步内部发生了啥"
              ▼
③Logging  拿 traceId 搜日志："支付服务那条日志写着 数据库连接池耗尽"
              │
              ▼
            根因找到 ✅
```

一句话记住三者的角色：

| 支柱 | 回答的问题 | 类比 |
|---|---|---|
| **Metrics** | 系统**整体**健康吗？有没有异常趋势？ | 体检报告上的各项数值 |
| **Tracing** | **这一次请求**经过了哪些环节，慢/错在哪一环？ | 一次就诊的完整流程单 |
| **Logging** | 那个出问题的环节**内部具体**发生了什么？ | 医生写的详细诊断记录 |

它们靠 **traceId 这根线**穿起来：指标发现异常 → 跳到对应的 trace 看链路 → 用 traceId 下钻到日志看细节。三层打通，排查效率是质的飞跃。

## 10. 小结

| 主题 | 关键要点 |
|---|---|
| 为什么需要 | 微服务把一个请求打散到多个进程，传统"翻一个日志、打两个时间戳"的排查全失灵 |
| 核心思想 | 给每个请求发一个全局唯一 **TraceId**（像快递单号），跟着请求一路传，把碎片串成线 |
| Span | 一段有起止时间的操作，带 spanId / traceId / parentSpanId |
| Trace | 一次请求的所有 Span 按父子关系拼成的一棵树，画出来就是甘特图 |
| 上下文传播 | 进程内靠 **ThreadLocal**（线程口袋）；跨服务靠把 traceId 塞进 **HTTP 头**（W3C `traceparent`） |
| 数据收集 | 自动埋点（Java Agent 字节码增强）产生 Span → 采样 → 异步上报给 Collector → 存储 → UI |
| 采样 | 头部采样简单省资源，尾部采样能保住出错链路；本质是成本 vs 完整性的权衡 |
| 生态 | OpenTelemetry 是统一标准（门面思想），后端可选 Jaeger / Zipkin / SkyWalking |
| 落地 | Spring Boot 用 Micrometer Tracing，自动埋点为主；关键是让**日志带上 traceId** |
| 常见坑 | 异步/线程池丢上下文、跨机器时钟漂移、采样可能漏掉关键请求 |
| 三大支柱 | Metrics 发现异常 → Tracing 定位环节 → Logging 看细节，靠 traceId 串通 |

链路追踪听起来高大上，剥开看核心就一句话：**给请求发个单号，每一跳都带着它、并记下自己这段的起止时间，最后按单号拼成一棵树。** 想清楚这条主线，剩下的采样、上报、各种工具，都只是围着它做工程优化而已。

> 到这里，可观测性三大支柱——[指标](/2026/05/05/Java-advanced/05-metric/)、[Grafana 大盘](/2026/05/06/Java-advanced/06-grafana/)、链路追踪——就凑齐了。实际项目里它们往往整合在同一套平台（比如在 Grafana 里从一个变慢的指标点，直接跳到对应的 trace，再下钻到日志），三层联动，排查一气呵成。

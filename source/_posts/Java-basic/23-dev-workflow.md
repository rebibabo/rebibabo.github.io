## 前言

写代码只是软件开发中的一个环节。入职后你会发现，真正的工作流程是：需求评审 → 技术方案 → 分支开发 → Code Review → 测试 → 部署 → 监控。这篇文章把一个需求从"产品经理提出"到"上线被用户使用"的完整链路走一遍，帮你在入职前建立全局视角。

<!-- more -->

## 1. 全景概览

一个需求的典型生命周期：

```
产品调研 → 需求文档 → 需求评审 → 技术方案 → 排期
    → 分支开发 → 自测 → Code Review → 合并
    → 测试环境部署 → QA 测试 → Bug 修复
    → 预发布环境验证 → 上线审批 → 生产部署
    → 线上监控 → 复盘
```

不同公司流程细节有差异，但主干基本一致。下面逐步展开。

## 2. 需求阶段

### 2.1 产品调研

产品经理（PM）负责：

```
市场/用户调研
  → 确定要解决的问题
  → 竞品分析
  → 产出 PRD（Product Requirements Document，产品需求文档）
```

PRD 通常包含：背景与目标、用户故事、功能清单、交互原型图（Figma/Axure）、优先级、验收标准。

### 2.2 需求评审

PM 召集所有相关人开会过 PRD：

```
参与者：PM、前端、后端、测试、设计师、项目经理
目标：对齐理解，发现遗漏，评估可行性

后端需要关注的：
├── 这个功能涉及哪些服务和接口？
├── 数据结构需要怎么设计？
├── 有没有性能、并发、安全方面的风险？
├── 和现有系统的哪些模块有交互？
└── 有没有不合理的需求要推回去？
```

评审后，PM 修改 PRD，产出最终版本。

### 2.3 技术方案

后端开发者输出技术方案文档（Tech Design Doc），通常包含：

```
1. 需求概述（一两句话说清楚要做什么）
2. 整体架构（系统交互图、时序图）
3. 数据库设计（ER 图、建表 SQL、索引设计）
4. 接口设计（API 列表、请求/响应格式）
5. 核心逻辑（算法、状态机、并发处理方案）
6. 缓存策略（哪些数据缓存、TTL、失效方案）
7. 风险与降级方案（如果依赖的服务挂了怎么办）
8. 排期（按子任务拆分，估算工时）
```

技术方案需要组内评审（Tech Review），资深工程师或架构师把关。

### 2.4 排期

```
常见的估算方式：
├── 乐观估计 × 1.5 = 实际排期（给自测和 buffer 留时间）
├── 按子任务拆分，每个任务 0.5-2 天
└── 包含自测时间（通常占开发时间的 30-50%）

一个典型的排期：
需求评审 + 技术方案：  2 天
开发：                 5 天
自测 + Code Review：   2 天
联调（前后端对接）：    1 天
QA 测试：              3 天
Bug 修复：             1 天
预发布验证 + 上线：     1 天
总计：                 约 15 天（3 周）
```

## 3. 开发阶段

### 3.1 Git 分支管理

大多数团队使用 Git Flow 或其变体：

```
main (master)         ← 生产环境代码，始终可发布
  │
  ├── develop         ← 开发主线，集成所有已完成的功能
  │     │
  │     ├── feature/add-frequency-cap    ← 功能分支（从 develop 切出）
  │     ├── feature/budget-control       ← 另一个功能分支
  │     └── ...
  │
  ├── release/v1.2.0  ← 发布分支（从 develop 切出，只修 bug）
  │
  └── hotfix/fix-crash ← 紧急修复分支（从 main 切出）
```

日常开发流程：

```bash
# 1. 从 develop 切出功能分支
git checkout develop
git pull origin develop
git checkout -b feature/add-frequency-cap

# 2. 在功能分支上开发，频繁提交
git add .
git commit -m "feat: 实现广告频控基础逻辑"
git commit -m "feat: 添加 Redis 频控计数器"
git commit -m "test: 补充频控单元测试"

# 3. 推送到远程
git push origin feature/add-frequency-cap

# 4. 在 GitHub/GitLab 上创建 Pull Request (PR) / Merge Request (MR)
#    → 关联需求单号
#    → 指定 Reviewer

# 5. Review 通过后合并到 develop
# 6. 删除功能分支
```

### 3.2 Commit 规范

```bash
# 格式：<type>: <subject>
feat: 新增广告频控功能
fix: 修复竞价超时未正确处理的问题
refactor: 重构出价策略为策略模式
test: 补充 BidService 单元测试
docs: 更新 API 文档
chore: 升级 Spring Boot 版本到 3.2.1
perf: 优化批量查询广告位的 SQL

# 常用 type：
# feat     新功能
# fix      修复 bug
# refactor 重构（不改变功能）
# test     测试
# docs     文档
# chore    构建/依赖/配置
# perf     性能优化
# style    代码格式（不影响功能）
```

### 3.3 开发规范

```
代码规范：
├── 遵循团队的 Code Style（通常有 checkstyle / spotless 插件自动检查）
├── 类名、方法名、变量名有意义（不要 a, b, temp）
├── 方法不超过 50 行，类不超过 500 行（超了就该拆分）
├── 注释写"为什么"，不写"是什么"（代码本身应该能说明"是什么"）
└── 不要提交注释掉的代码、System.out.println、硬编码的密码

接口规范：
├── RESTful URL 设计（名词复数、HTTP 方法语义正确）
├── 统一的响应格式（code + message + data）
├── 统一的错误码体系
├── 接口文档（Swagger / OpenAPI）
└── 版本控制（/api/v1/...）
```

### 3.4 自测

代码写完不是直接提交 Review，而是先自测：

```
自测 checklist：
├── 单元测试全部通过
├── 新功能有对应的单元测试（行覆盖率达到团队要求）
├── 本地启动服务，用 Postman / curl 验证接口
├── 正常流程跑通
├── 异常流程验证（参数为空、不存在的 ID、超时、并发）
├── 边界情况（空列表、超长字符串、最大值最小值）
└── 日志输出正确，没有敏感信息泄露
```

## 4. Code Review

### 4.1 为什么要 Code Review？

```
发现 bug         ← 另一双眼睛能看到你看不到的问题
知识共享          ← 团队成员了解彼此的代码
代码质量          ← 保持一致的风格和设计水平
新人成长          ← 通过 Review 学习团队的最佳实践
```

### 4.2 提交 PR 的规范

```markdown
## PR 标题
feat: 实现广告频控功能

## 描述
- 需求背景：每个用户每天看同一广告最多 N 次
- 实现方案：Redis INCR + 当天过期
- 关联需求：JIRA-1234

## 改动范围
- 新增 FrequencyCapService
- 新增 FrequencyCapServiceTest
- 修改 BidService，在竞价前检查频控

## 测试情况
- 单元测试：✅ 全部通过
- 本地联调：✅ 正常流程 + 超限拒绝
```

### 4.3 Review 的关注点

```
Reviewer 通常会看：
├── 逻辑正确性：边界情况、并发安全、异常处理
├── 设计合理性：职责划分、是否过度设计或设计不足
├── 可读性：命名、注释、方法长度
├── 性能：有没有 N+1 查询、不必要的循环、大对象拷贝
├── 安全性：SQL 注入、XSS、敏感信息暴露
├── 测试覆盖：关键逻辑有没有测试
└── 规范遵守：命名规范、日志规范、接口规范
```

### 4.4 Review 的礼仪

```
作为 Reviewer：
├── 对事不对人，语气友善
├── 说明"为什么"而不只是"改成这样"
├── 区分 must-fix 和 nice-to-have
└── 大段重构建议线下沟通，PR 上说清结论即可

作为被 Review 者：
├── 不要把 Review 意见当批评
├── 有争议的点可以讨论，但不要情绪化
├── 复杂逻辑主动加注释或在 PR 描述中解释
└── 及时回复和修改，不要让 PR 挂太久
```

## 5. 测试阶段

### 5.1 测试环境

```
一般至少有 3 套环境：

开发环境 (DEV)
  ├── 开发自测用
  ├── 数据随意
  └── 部署频率高（可能一天几十次）

测试环境 (QA / Staging)
  ├── QA 团队测试用
  ├── 有稳定的测试数据
  └── 功能分支合并到 develop 后自动部署

预发布环境 (Pre-production)
  ├── 配置和生产完全一致（连同一个数据库或其只读副本）
  ├── 上线前最后验证
  └── 只有即将上线的代码才部署到这里
```

### 5.2 测试类型

| 类型 | 谁做 | 工具 | 关注点 |
|------|------|------|--------|
| 单元测试 | 开发 | JUnit + Mockito | 方法级别的逻辑正确性 |
| 集成测试 | 开发 | @SpringBootTest | 多模块协同、数据库交互 |
| 接口测试 | QA | Postman / JMeter | API 的输入输出、状态码 |
| 功能测试 | QA | 手动 / Selenium | 业务流程端到端验证 |
| 回归测试 | QA | 自动化脚本 | 新功能有没有破坏老功能 |
| 性能测试 | 开发/QA | JMeter / wrk | 吞吐量、延迟、资源占用 |

### 5.3 Bug 管理

```
QA 在 JIRA / Tapd / Zentao 中提 Bug：
├── 标题：简洁描述问题（"频控计数在跨天时未重置"）
├── 优先级：P0（阻塞）/ P1（严重）/ P2（一般）/ P3（优化）
├── 重现步骤：具体的操作路径
├── 期望结果 vs 实际结果
├── 环境信息：浏览器/设备/测试账号
└── 截图或日志

Bug 修复流程：
开发修复 → 自测 → 提交到 develop → 部署到测试环境 → QA 回验 → 关闭 Bug
```

## 6. 部署阶段

### 6.1 CI/CD 流水线

持续集成（CI）和持续部署（CD）自动化了从提交代码到部署的全过程：

```
开发者 push 代码
     │
     ▼
CI 流水线自动触发（Jenkins / GitLab CI / GitHub Actions）
     │
     ├── 1. 代码检查（Lint / Checkstyle）
     ├── 2. 编译
     ├── 3. 运行单元测试
     ├── 4. 运行集成测试
     ├── 5. 构建 Docker 镜像
     ├── 6. 推送镜像到镜像仓库
     └── 7. 部署到对应环境
          ├── develop 分支 → 自动部署到测试环境
          ├── release 分支 → 部署到预发布环境
          └── main 分支 → 部署到生产环境（通常需要人工审批）
```

### 6.2 Docker 部署

```dockerfile
# Dockerfile
FROM eclipse-temurin:17-jre
WORKDIR /app
COPY build/libs/ssp-service-1.0.0.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar", \
    "-Xms512m", "-Xmx512m", \
    "--spring.profiles.active=prod"]
```

```yaml
# docker-compose.yml（简单场景）
services:
  ssp-service:
    image: ssp-service:1.0.0
    ports:
      - "8080:8080"
    environment:
      - SPRING_PROFILES_ACTIVE=prod
      - SPRING_DATASOURCE_URL=jdbc:mysql://db:3306/ssp
    depends_on:
      - db
      - redis

  db:
    image: mysql:8.0
    environment:
      - MYSQL_ROOT_PASSWORD=secret
      - MYSQL_DATABASE=ssp

  redis:
    image: redis:7
```

### 6.3 发布策略

| 策略 | 做法 | 优点 | 风险 |
|------|------|------|------|
| 全量发布 | 所有实例同时更新 | 简单 | 出问题影响全部用户 |
| **灰度发布**（推荐） | 先更新 10% 的实例，观察正常后再全量 | 风险可控 | 需要基础设施支持 |
| 蓝绿部署 | 两套环境，切流量到新版本 | 秒级回滚 | 需要双倍资源 |
| 金丝雀发布 | 新版本只给少量用户（按用户 ID 或地区） | 精细控制 | 实现复杂 |

```
灰度发布流程（大厂标准做法）：
1. 部署新版本到 10% 的实例
2. 观察 5-15 分钟：错误率、延迟、日志
3. 没问题 → 扩大到 50%
4. 再观察 → 全量
5. 有问题 → 立即回滚
```

### 6.4 数据库变更

```
代码可以秒级回滚，但数据库变更不能。所以数据库变更需要特别谨慎：

原则：
├── 向前兼容：新代码能用旧表结构，旧代码能用新表结构
├── 只加不删：先加新列 → 代码迁移 → 确认旧列无引用后再删
├── 大表加索引要在低峰期（会锁表）
├── 变更前先在测试环境验证
└── 使用数据库迁移工具（Flyway / Liquibase）管理变更版本

Flyway 示例：
src/main/resources/db/migration/
├── V1__create_users_table.sql
├── V2__add_email_column.sql
└── V3__create_orders_table.sql
```

## 7. 上线后

### 7.1 监控

```
上线不是终点，而是新的开始。需要持续监控：

基础设施监控（Prometheus + Grafana）：
├── CPU / 内存 / 磁盘 / 网络
├── JVM 堆内存、GC 次数和耗时
└── 线程池活跃线程数

应用监控（Spring Boot Actuator + Micrometer）：
├── QPS / 请求延迟（P50 / P99）
├── 错误率（4xx / 5xx）
├── 接口响应时间
└── 自定义业务指标（竞价成功率、广告填充率）

日志监控（ELK：Elasticsearch + Logstash + Kibana）：
├── ERROR 日志告警
├── 异常堆栈分析
└── 请求链路追踪（TraceId）

告警：
├── 错误率 > 1% → 通知到企业微信/钉钉群
├── P99 延迟 > 500ms → 通知
├── 服务实例挂了 → 电话告警
└── 磁盘使用率 > 80% → 通知
```

### 7.2 线上问题排查

```
收到告警后的标准动作：

1. 先止血
   ├── 能回滚就回滚（新版本引入的问题）
   ├── 不能回滚就降级（关掉出问题的功能、切到兜底逻辑）
   └── 扩容（如果是流量暴增导致）

2. 再定位
   ├── 看监控：哪个指标异常？什么时候开始？
   ├── 看日志：grep 错误日志，找堆栈
   ├── 看变更：最近谁上线了什么？有没有配置变更？
   └── 看依赖：下游服务是不是挂了？数据库是不是慢了？

3. 最后修复
   ├── 修复代码 → 走正常发布流程
   └── 紧急情况 → hotfix 分支 → 快速 Review → 灰度上线
```

### 7.3 复盘

重要功能上线后或出现线上事故后，团队会做复盘：

```
复盘内容：
├── 时间线：什么时候发现、什么时候处理、什么时候恢复
├── 根因分析：为什么出问题？（5 Whys 方法）
├── 影响范围：影响了多少用户？持续了多久？
├── 改进措施：怎么防止同类问题再次发生？
│   ├── 代码层面：补充测试、加入校验
│   ├── 流程层面：增加 Review 环节、完善 checklist
│   └── 基础设施：加监控、加告警
└── 跟进事项：指定责任人和完成时间
```

## 9. 新人入职第一周生存指南

```
Day 1：
├── 领电脑、开通各种权限（Git、JIRA、VPN、数据库）
├── 拉代码、搭建本地开发环境
└── 跑通项目（能本地启动、能调通接口）

Day 2-3：
├── 阅读项目 README 和技术文档
├── 阅读代码：从入口（Controller）开始，跟一个核心链路
├── 了解项目的模块划分和分层架构
└── 梳理数据库表结构

Day 4-5：
├── 领一个小需求或 Bug（通常 mentor 会安排）
├── 写代码 → 自测 → 提 PR → 接受 Review
├── 观察其他人的 PR 是怎么写的
└── 记录遇到的问题，不懂就问（有 mentor 就问 mentor）

心态：
├── 第一周看不懂代码是正常的，项目都是几十万行起
├── 不要怕问问题，但先自己搜 / 查文档 / 看代码 15 分钟
├── commit 要小而频繁，不要攒一大坨
└── 主动记录学到的东西，写成笔记
```

## 10. 小结

| 阶段 | 关键动作 | 核心产出 |
|------|---------|---------|
| 需求 | 评审 PRD、确认技术可行性 | 技术方案文档 |
| 开发 | 切分支、写代码、写测试、自测 | 功能代码 + 单元测试 |
| Review | 提 PR、接受反馈、修改 | 通过 Review 的代码 |
| 测试 | QA 验证、修 Bug、回归 | 测试报告 |
| 部署 | CI/CD 构建、灰度发布 | 线上运行的服务 |
| 监控 | 观察指标、处理告警 | 稳定运行的系统 |
| 复盘 | 总结问题、制定改进措施 | 复盘文档 |

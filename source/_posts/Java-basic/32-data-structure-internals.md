## 前言

[03 集合框架](/2026/05/03/Java-basic/03-collections/) 讲的是"怎么用"——`ArrayList`、`HashMap` 的 API。这篇讲"为什么这么设计"——也就是**底层原理**，重点是 `HashMap`，因为它几乎是 Java 面试出现频率最高的话题。

注意范围：这里说的"数据结构底层原理"，指的是**Java 集合类的内部实现**（数组怎么扩容、HashMap 的哈希和扩容机制、为什么用红黑树），**不是**让你去手写红黑树、AVL树这类算法题——那是另一个范畴，工作中很少需要自己实现，但面试官很爱问"HashMap 底层用了什么数据结构、为什么"。

## 1. ArrayList 底层：数组与扩容

`ArrayList` 底层是一个 `Object[]` 数组。我们都知道数组创建后长度固定，那 `ArrayList` 是怎么做到"可以一直 `add`"的？

### 1.1 扩容机制

```java
List<String> list = new ArrayList<>(); // 默认容量 10（第一次 add 时才真正分配数组）
```

当数组满了，再 `add` 时会触发扩容：

1. 创建一个**新数组**，容量是旧容量的 **1.5 倍**（`oldCapacity + oldCapacity >> 1`）
2. 把旧数组的所有元素 `System.arraycopy` 复制到新数组
3. 新元素放进新数组，原数组变成垃圾，等待 GC

```
扩容前: [10个元素] (容量10，已满)
add 第11个元素 →
扩容后: [11个元素 + 4个空位] (容量 10 + 10>>1 = 15)
```

**为什么是 1.5 倍，不是 2 倍或者每次只加1？**

- 如果每次只扩容1个，每次 `add` 几乎都要触发"创建新数组+整体复制"，时间复杂度退化为 O(n)
- 如果扩容倍数太大（比如2倍），容易造成内存浪费，尤其是大数组场景
- 1.5 倍是时间和空间的折中——这也是为什么如果你**提前知道大概要存多少元素**，最好用 `new ArrayList<>(initialCapacity)` 指定初始容量，**避免多次扩容复制**

### 1.2 为什么 `add` 的"均摊"时间复杂度是 O(1)

虽然扩容这一次操作是 O(n)（要复制所有元素），但扩容不是每次都发生——大部分 `add` 是 O(1)（直接放到数组末尾）。把扩容的成本"摊"到多次普通的 `add` 上，平均下来每次 `add` 仍然是 O(1)，这就是**均摊时间复杂度（Amortized Time Complexity）**的概念。

### 1.3 `remove`/`add(index, element)` 为什么慢

往中间插入或删除元素，需要把后面所有元素整体往前/往后移动一位（`System.arraycopy`），这是 O(n) 操作——这也是为什么频繁在中间增删时，`LinkedList` 理论上更合适（但实际场景要结合下面第7节再看）。

## 2. HashMap 底层：核心重点

### 2.1 整体结构

`HashMap` 的底层是一个**数组**，数组里的每个位置叫一个"桶（bucket）"，每个桶里存的是一个**链表**（JDK 8 之后，链表过长会转成**红黑树**）。

```
数组下标:  0     1        2          3
          │     │        │          │
桶内结构:  null  Node ──► Node──►Node  红黑树（节点数 > 8）
              (key1)   (key2) (key3)
```

为什么会有"桶内多个元素"——因为不同的 key，经过哈希计算后**可能落到同一个数组下标**，这种现象叫**哈希冲突**，链表/红黑树就是用来解决冲突的。

### 2.2 put 的完整流程

```java
map.put("Alice", 90);
```

1. 计算 `"Alice"` 的 `hashCode()`，再经过 `hash()` 函数做一次"扰动"，得到最终的 hash 值
2. 用 `hash & (capacity - 1)` 计算出数组下标（桶的位置）
3. 如果这个桶是空的，直接放进去
4. 如果桶里已经有元素了（哈希冲突）：
   - 逐个比较 `key` 的 `hashCode` 和 `equals`，如果找到相同的 key，**覆盖**旧值
   - 没找到相同的，把新节点**加到链表末尾**（JDK 8 改为尾插，JDK 7 是头插）
5. 如果链表长度超过 8，且数组容量 ≥ 64，链表转换成**红黑树**
6. 插入后，检查元素总数是否超过 `容量 × 负载因子`（默认 0.75），超过就触发**扩容**

### 2.3 hash() 扰动函数：为什么不直接用 hashCode

```java
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

这一行代码做的事：把 `hashCode()` 的**高16位**和**低16位**做异或，得到新的 hash 值。

**为什么要这么做？**

数组下标是用 `hash & (capacity - 1)` 算出来的。如果 `capacity` 比较小（比如默认16，二进制 `10000`，减1是 `01111`），`hash & 15` 实际上**只用到了 hash 值的低4位**——高位的信息完全被忽略了。

如果很多 key 的 `hashCode()` 高位不同但低位相同（这种情况并不少见），它们会全部落入同一个桶，造成严重的哈希冲突。

扰动函数把高16位"混入"低16位，让**高位的差异也能影响最终的下标**，让哈希分布更均匀。这是用很小的计算成本（一次异或），换取明显更好的分布效果。

### 2.4 为什么数组长度必须是 2 的幂

`hash & (capacity - 1)` 这个位运算，**只有当 `capacity` 是 2 的幂时，才等价于 `hash % capacity`**，但位运算比取模运算快得多。

```
capacity = 16 (二进制 10000)
capacity - 1 = 15 (二进制 01111)
hash & 15  ←→  hash % 16   (结果完全一样，但位运算更快)
```

如果 capacity 不是2的幂，`hash & (capacity-1)` 就不再等价于取模，会导致某些下标永远取不到，分布严重不均。这也是为什么即使你 `new HashMap<>(18)`，HashMap 内部也会自动调整为大于18的最小的2的幂——也就是32。

### 2.5 负载因子 0.75：为什么不是 1 或者 0.5

负载因子（Load Factor）决定了"数组用到多满就扩容"——`阈值 = 容量 × 负载因子`，默认 `16 × 0.75 = 12`，也就是放入第13个元素时触发扩容。

| 负载因子 | 空间利用率 | 哈希冲突概率 | 综合评价 |
|---|---|---|---|
| 接近 1（比如 1.0） | 高（数组几乎装满才扩容） | 高（桶里链表会很长，查询变慢） | 省空间但费时间 |
| 接近 0（比如 0.5） | 低（数组经常是半空的，浪费内存） | 低 | 费空间但省时间 |
| **0.75** | 中等 | 中等 | **时间和空间的折中**，是大量统计实验后得出的经验值 |

### 2.6 扩容（resize）：容量翻倍 + 重新计算位置

扩容时，数组容量变成**原来的2倍**（比如16→32）。容量变了，意味着 `hash & (capacity-1)` 这个公式的结果也会变——所以**所有元素都要重新计算一遍它在新数组里的位置**，这个过程叫 **rehash**。

```
扩容前: capacity=16, hash & 15
扩容后: capacity=32, hash & 31   ← 多用了一位，每个元素的新下标只有两种可能：原下标，或者"原下标 + 16"
```

这个特性（JDK 8 的优化）让 rehash 不需要重新计算 hash 值，只需要看 hash 值的"新增的那一位"是0还是1，就知道元素留在原位还是移到"原下标+oldCapacity"，效率比 JDK 7 的全量重新计算高很多。

### 2.7 链表转红黑树：为什么阈值是 8

| 链表长度 | 数据结构 | 查询时间复杂度 |
|---|---|---|
| ≤ 8 | 链表 | O(n)，但 n 很小，链表的简单遍历反而比红黑树更快（红黑树有额外的旋转、平衡开销） |
| > 8（且数组容量 ≥ 64） | 红黑树 | O(log n) |

**为什么是 8 这个数字？** JDK 源码注释里提到，在 hash 值分布良好的情况下，一个桶里链表长度达到 8 的概率，按泊松分布计算大约是 **百万分之六**——也就是说，**正常情况下几乎不会触发树化**。如果某个桶里链表长度真的达到了8，大概率意味着 `hashCode()` 设计得不好（哈希冲突严重），这时候用红黑树兜底，把最坏情况的查询复杂度从 O(n) 降到 O(log n)。

> 还有一个细节：树退化回链表的阈值是 **6**，不是 8——这是为了避免链表长度在 7、8 之间反复增减时，频繁地"树化→退化→树化"，造成性能抖动（这种为避免临界值反复抖动而设置"缓冲区间"的设计思路，在很多系统里都能看到）。

## 3. equals 与 hashCode 的契约

`HashMap` 能正常工作的前提，是 key 类正确重写了 `equals()` 和 `hashCode()`，而且必须**同时重写、保持一致**。

### 3.1 契约内容

| 规则 | 说明 |
|---|---|
| 如果两个对象 `equals()` 返回 `true` | 它们的 `hashCode()` **必须相同** |
| 如果两个对象 `hashCode()` 相同 | 它们的 `equals()` **不一定**要相同（哈希冲突是允许的，链表就是用来处理这种情况的） |
| 如果只重写 `equals()`，不重写 `hashCode()` | **违反契约**——两个逻辑相等的对象，可能因为 `hashCode()` 不同被分到不同的桶，HashMap 会认为它们是"不同的key" |

### 3.2 一个具体的坏例子

```java
class User {
    String name;
    @Override
    public boolean equals(Object o) {
        return o instanceof User && ((User) o).name.equals(this.name);
    }
    // 没有重写 hashCode()，用的是 Object 默认的（基于内存地址）
}

Map<User, String> map = new HashMap<>();
map.put(new User("Alice"), "数据1");
map.get(new User("Alice")); // 返回 null！

// 原因：两个 new User("Alice") 的 equals() 为 true，但 hashCode() 不同（默认基于地址）
// 所以它们被分到了不同的桶，get 时根本找不到 put 时存的那个桶
```

**记忆方式**：`hashCode()` 决定"去哪个桶里找"，`equals()` 决定"桶里哪个才是你要的那个"。如果 `hashCode()` 不一致，第一步"去哪个桶"就错了，第二步根本不会执行。

## 4. HashMap 的线程安全问题

`HashMap` **不是线程安全的**，多线程并发 `put` 可能导致：

| JDK 版本 | 问题 | 原因 |
|---|---|---|
| JDK 7 | **resize 时可能死循环**，CPU 飙到100% | JDK 7 扩容时链表是**头插法**，多线程同时扩容可能导致链表中出现环形引用 |
| JDK 8 | 不会死循环，但仍可能**数据覆盖/丢失** | 多线程同时判断"这个桶是空的"，都执行了"放入新节点"，后写的会覆盖先写的 |

**结论**：多线程场景下，不要用 `HashMap`，要用 `ConcurrentHashMap`。`HashMap` 本身的设计目标就是单线程下的高性能，线程安全交给专门的并发集合处理。

## 5. ConcurrentHashMap：线程安全的 HashMap

| JDK 版本 | 并发方案 |
|---|---|
| JDK 7 | **分段锁（Segment）**——把数据分成16个段，每段独立加锁，不同段的操作可以并行 |
| JDK 8 | **CAS + synchronized**——put 时先用 CAS 尝试无锁插入空桶；如果桶不为空（有冲突），只对**这个桶的头节点**加 `synchronized` 锁，锁粒度精确到单个桶，而不是整个 Map 或整个段 |

**为什么 JDK 8 要改成这种方式？** 分段锁的"段数"是固定的（默认16），并发度的上限就是16。JDK 8 把锁的粒度细化到"每个桶"，理论上并发度等于数组长度（默认16，但会随扩容增长），并发能力更高，且去掉了 Segment 这一层额外的内存开销。

实际开发中，`ConcurrentHashMap` 的用法和 `HashMap` 完全一样（都实现了 `Map` 接口），唯一区别就是"多线程环境下用 `ConcurrentHashMap`"。

## 6. LinkedList vs ArrayDeque：为什么官方推荐后者做栈/队列

`LinkedList` 是双向链表，`ArrayDeque` 是基于数组的双端队列。两者都实现了 `Deque` 接口，都能当栈/队列用，但 JDK 文档明确建议：**"作为栈或队列，`ArrayDeque` 比 `LinkedList` 更快"**。

| 维度 | LinkedList | ArrayDeque |
|---|---|---|
| 底层结构 | 每个元素是一个独立对象（Node），包含 prev/next 指针 | 连续的数组 |
| 内存开销 | 每个元素额外有2个指针的开销（约16字节/元素） | 几乎无额外开销 |
| 缓存友好性 | 差——链表节点在内存中分散，CPU缓存命中率低 | 好——数组连续存储，符合CPU缓存的"局部性原理" |
| null 元素 | 允许 | **不允许**（`null` 被用作"队列为空"的特殊返回值） |

**为什么数组反而比链表快？** 现代CPU访问内存时会按"缓存行"批量加载数据，连续存储的数组能更好地利用这一特性（访问 `arr[i]` 时，`arr[i+1]` 大概率已经被加载进缓存）；而链表的节点分散在堆内存各处，每访问一个节点都可能是一次"缓存未命中"，需要重新从内存读取。

**所以**：除非你确实需要"在链表中间频繁插入/删除"这种场景，否则栈/队列优先用 `ArrayDeque`；`LinkedList` 在现代 Java 项目中已经比较少见了。

## 7. TreeMap/TreeSet：红黑树

`TreeMap`/`TreeSet` 底层是**红黑树**——一种自平衡的二叉搜索树，保证增删查的时间复杂度都是 O(log n)，且遍历时 key 始终有序。

### 7.1 为什么是红黑树，不是普通二叉搜索树或 AVL 树

| 数据结构 | 问题/特点 |
|---|---|
| 普通二叉搜索树 | 如果插入的数据正好是有序的（比如依次插入1,2,3,4,5），会退化成一条链表，查询变成 O(n) |
| AVL 树 | 严格平衡（左右子树高度差≤1），查询效率略高于红黑树，但**插入/删除时为了维持严格平衡，旋转操作更频繁** |
| **红黑树** | 近似平衡（不要求严格的高度差），插入/删除的**旋转次数更少**，整体的增删查综合性能更好 |

**一句话**：AVL树"读多写少"场景更优；红黑树在"读写均衡"的场景更优。Java 集合（`TreeMap`/`TreeSet`，以及 `HashMap` 的树化桶）选择了红黑树，是因为通用场景下增删操作的频率不可忽视。

## 8. 小结

| 主题 | 核心要点 |
|---|---|
| ArrayList扩容 | 容量不足时创建1.5倍大小的新数组并整体复制；预知数据量时指定初始容量可避免多次扩容 |
| HashMap整体结构 | 数组(桶) + 链表，链表长度>8且数组容量≥64时转红黑树 |
| hash()扰动函数 | 高16位异或低16位，让高位信息也能影响数组下标，使分布更均匀 |
| 数组长度为2的幂 | 使 `hash & (capacity-1)` 等价于取模，且位运算比取模快 |
| 负载因子0.75 | 时间(冲突率)和空间(利用率)的折中 |
| 扩容机制 | 容量翻倍，rehash时元素新位置只有"原位置"或"原位置+oldCapacity"两种可能 |
| 树化阈值8/退化阈值6 | 8是基于泊松分布得出的"几乎不会触发"的安全值；8和6之间留缓冲避免反复树化 |
| equals/hashCode契约 | 必须同时重写且保持一致，否则 `get` 找不到 `put` 进去的元素 |
| HashMap线程不安全 | JDK7扩容可能死循环，JDK8可能数据丢失，多线程用ConcurrentHashMap |
| ConcurrentHashMap | JDK7分段锁(16段)，JDK8 CAS+按桶加synchronized锁，并发度更高 |
| ArrayDeque vs LinkedList | 数组的内存连续性对CPU缓存更友好，栈/队列场景官方推荐ArrayDeque |
| TreeMap红黑树 | 近似平衡，旋转次数比AVL树少，增删查综合性能更优 |

这些内容里，**HashMap 的 hash 扰动、负载因子、扩容机制、树化阈值、equals/hashCode 契约**是面试被问到概率最高的几个点，建议重点掌握；`ConcurrentHashMap` 的两代实现和 `ArrayDeque` 的缓存友好性属于"加分项"，能说出来会显得基础更扎实。
